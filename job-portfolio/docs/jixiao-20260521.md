## 一、制度演进总览

### 1.1 时间线

| 版本 | 时间 | 性质 | 核心变化 |
|------|------|------|----------|
| V1 初始版 | 2024年1月 | 奠基 | 四大维度框架确立：交付60%+质量20%+技术沉淀15%+其他5% |
| V2 第一次调整 | 2024年11月 | 精细化 | 引入弹性封顶评分、降低固定加分、强化反讲机制、项目跟进人自荐制 |
| V2修正 | 2024年12月 | 微调 | 与11月版基本一致 |
| V3 第二次调整 | 2025年2月 | 流程完善 | 新增发布工作主导者奖励 |
| V4 第三次调整 | 2025年4月 | 质量深化 | 新增月度测试质量、探索型任务、代码质量CodeView |

### 1.2 底层框架（始终未变）

```
总分100分 = 交付(60分) + 质量(20分) + 技术沉淀(15分) + 其他(5分)
```

这个框架从V1到V4从未动摇，体现设计者对"工程师核心价值"的清晰判断：
- **交付是硬道理**（60%权重）— 产出是第一位的
- **质量是生命线**（20%权重）— 不能为了快而牺牲质量
- **技术沉淀是团队资产**（15%权重）— 个人成长反哺团队
- **其他是基础纪律**（5%权重）— 考勤、同步、沟通

## 二、各维度演进详情

### 2.1 交付维度（60分）

#### 2.1.1 有效工时评分（40分）

| 版本 | 评分机制 | 关键特征 |
|------|----------|----------|
| V1~V4 | 组内中位数对比 + 调节基准 | 始终未变 |

**调节基准公式**（全版本一致）：
```
基准 = 有效工作日 × 7工时/天 × 70%
```
- 各组锚点：网站组/前端组/Java组均以70%为锚点
- 计分规则：中位数和调节基准取高者
- 请假时间不剔除（人性化设计）

#### 2.1.2 交付及时性（20分）

| 版本 | 需求分级 | 项目跟进人机制 | 扣分系数 |
|------|----------|--------------|----------|
| V1 (24Q1) | 简单<2人天 / 中等2~5人天 / 复杂>1人周 | 首要人×1 + 负责人×0.5 + 成员×0.3（取最大） | 提测/上线分别计算 |
| V2 (24Q4) | 简单<2人天 / 中等2~5人天 / 复杂>5人天 | **项目跟进人×1 + 成员×0.5**（简化） | 提测/上线分别计算 |
| V2.1 (24Q4修正) | 同V2 | 同V2 | 同V2 |
| V3 (2502) | 同V2 | **新增**：先自荐，无自荐则开发评审会指定 | 同V2 |
| V4 (2504) | 同V2 | 同V3 | 同V2 |

**关键演进：从"三级责任链"到"两级责任制"**
- V1的"首要人/负责人/成员"过于复杂，实际操作中界定困难
- V2简化为"项目跟进人/成员"，降低了管理成本
- V3引入"自荐+指定"机制，既鼓励主动性又保证不落空

### 2.2 质量维度（20分）

#### 2.2.1 Bug修复及时率（10分）— 全版本一致

| Bug级别 | 定义 | 解决时限 | 扣分 |
|---------|------|----------|------|
| 1级 | 系统崩溃，彻底无法测试 | 0.5天 | 延迟天数×2分 |
| 2级 | 影响主流程测试 | 0.5天 | 延迟天数×1分 |
| 3级 | 不影响主流程的功能问题 | 1天 | 延迟天数×0.5分 |
| 4级 | 界面、性能等建议性质 | 可视排期 | 不扣 |

#### 2.2.2 二次Bug率（5分）— 全版本一致

| Bug级别 | 允许比率 | 扣分 |
|---------|----------|------|
| 1级 | 0% | 数量×反复次数×2分 |
| 2级 | 5% | 数量×反复次数×0.5分 |
| 3级 | 10% | 数量×反复次数×0.1分 |

> 二次Bug定义：同一个bug或场景，回归测试2次以上（含2次）
> 二次Bug率 = 二次Bug次数 / 项目人天

#### 2.2.3 当月遗留缺陷（5分）— 全版本一致

当月创建的BUG当月未解决即扣分，标准同Bug修复及时率。

#### 2.2.4 月度测试质量（2504新增）

**V4重大新增，分值-5到5分**

```
评估维度：
1. 当月测试bug数 ÷ 当月工时（剔除自测工时）= 时均bug数
2. 结合2级Bug数量，主管主观质量扣分
3. 主抓第1名和排名靠前且2级Bug数量较高者
4. 月度质量提升明显的，给予质量奖励分数
```

**设计理念**：从"个体bug考核"升级为"整体质量回顾"，引入主管主观判断权，防止开发者避重就轻只修小bug。

### 2.3 技术沉淀维度（15分）

这是**演进最剧烈**的维度，体现了从"固定激励"到"弹性封顶"、从"结果导向"到"过程+结果并重"的管理升级。

#### 2.3.1 知识沉淀（核心加分项）

| 加分项 | V1 (24Q1) | V2 (24Q4) | V3 (2502) | V4 (2504) |
|--------|-----------|-----------|-----------|-----------|
| 设计文档 | +2分/次 | **+1分/次（3分封顶）** | 同V2 | 同V2，注明"鼓励优秀" |
| 系统培训 | +2分/次 | **+1分/次（3分封顶）** | 同V2 | 同V2 |
| 代码走读分享 | +1分/次 | +1分/次 | +1分/次 | +1分/次 |
| 系统手册 | +2分/次 | **+1分/次（3分封顶）** | 同V2 | 同V2 |
| 项目跟进人加分 | 中等1分/复杂2分 | **中等0.5分/复杂1.5分** | 同V2 | **中等0.5分/复杂1分（3分封顶）** |
| 发布主导者 | — | — | **+0.5分/月封顶** | 同V3 |
| 探索型任务 | — | — | — | **1分起步，视重要程度加分（2504新增）** |
| 代码质量(CodeView) | — | — | — | **-3到3分，主管主观评分（2504新增）** |

#### 2.3.2 知识分享

| 加分项 | V1 | V2~V4 |
|--------|-----|-------|
| 小组内分享 | +1分/次 | +1分/次 |
| 单周论坛分享 | +2分/次，高质量+3分 | +2分/次，高质量+3分 |
| 复杂项目反讲 | +2分/次，高质量+3分 | **+1分起步（3分封顶）**，鼓励优秀 |

#### 2.3.3 演进分析：技术沉淀维度的三次升级

**第一次升级（V1→V2）：引入弹性封顶**
- 问题：V1固定+2分导致"刷文档"现象，有人写大量低质量文档刷分
- 解决：降为+1分，设3分封顶，引入"鼓励优秀"弹性
- 效果：从"数量导向"转为"质量导向"

**第二次升级（V2→V3）：增加流程奖励**
- 新增：发布主导者奖励0.5分/月
- 目的：激励DevOps流程 ownership，降低发布风险

**第三次升级（V3→V4）：引入过程质量和创新探索**
- 新增探索型任务：短期无法交付但必需的研究性任务，按迭代周期提交阶段性成果
- 新增代码质量(CodeView)：优秀代码+3分，整改不到位-3分
- 目的：从"只看产出"到"关注代码质量"和"鼓励技术创新"

## 三、绩效成长模型抽象

### 3.1 核心设计哲学

```
┌─────────────────────────────────────────────────────────────┐
│                    工程师绩效成长模型                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   第一层：结果导向（交付60%）                                  │
│   ├── 产出密度：有效工时评分（40分）                           │
│   └── 交付纪律：及时性（20分）                                │
│                                                             │
│   第二层：质量底线（质量20%）                                  │
│   ├── 响应速度：Bug修复及时率（10分）                          │
│   ├── 一次性做对：二次Bug率（5分）                            │
│   ├── 不遗留债务：当月遗留缺陷（5分）                          │
│   └── 【V4新增】整体质量回顾：月度测试质量（±5分弹性）           │
│                                                             │
│   第三层：团队资产（技术沉淀15%）                              │
│   ├── 文档化：设计文档、系统手册（封顶3分）                    │
│   ├── 知识传播：培训、分享、反讲（封顶3分）                    │
│   ├── 项目贡献：项目跟进人加分（封顶3分）                      │
│   ├── 流程 ownership：发布主导者（0.5分/月）                  │
│   ├── 【V4新增】创新探索：探索型任务（1分起步）                │
│   └── 【V4新增】代码质量：CodeView评分（-3到3分）              │
│                                                             │
│   第四层：基础纪律（其他5%）                                   │
│   ├── 信息同步（计划+禅道）                                   │
│   ├── 考勤纪律                                               │
│   └── 【规划】运维贡献                                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 演进规律总结

| 演进阶段 | 时间 | 核心特征 | 管理成熟度 |
|----------|------|----------|------------|
| **建立期** | 24Q1 | 框架确立、固定分值、简单直接 | L1 规则驱动 |
| **优化期** | 24Q4 | 引入弹性、降低刷分空间、细化规则 | L2 机制驱动 |
| **深化期** | 2502~2504 | 增加过程指标、引入主观判断权、鼓励创新 | L3 文化驱动 |

### 3.3 关键管理洞察

1. **权重骨架稳定，细则持续迭代**
   - 60/20/15/5的权重比例从未改变，说明设计者对这个比例有充分信心
   - 变化集中在"如何考核"而非"考核什么"

2. **从"量化一切"到"量化+定性结合"**
   - V1~V3：所有指标均可精确计算，追求客观
   - V4：引入"月度测试质量"（-5到5分主观分）和"代码质量"（-3到3分主观分）
   - 原因：纯量化指标容易被"优化"（刷分），需要主管判断力补充

3. **从"激励数量"到"激励质量"**
   - 文档/培训/手册加分从+2分降为+1分，封顶3分
   - 反讲从固定+2分变为+1分起步、3分封顶
   - 传递信号：不鼓励刷数量，鼓励真正有价值的产出

4. **从"结果考核"到"过程+结果"**
   - V4新增探索型任务：认可"没有即时产出但有长期价值"的工作
   - V4新增CodeView：把代码质量纳入考核，而不仅仅是bug数量

5. **数据源统一、减少争议**
   - 所有量化指标均有明确数据源：禅道系统、测试组、主管汇总
   - 工时计算规则明确（1人天=7工时，请假不剔除）
   - 有效减少"评分不公"的争议空间

## 五、对简历的价值提炼

### 5.1 可直接用于面试的话术

**关于绩效制度设计能力：**
> "我在团队推行了一套数据驱动的绩效考核体系，核心框架是交付60%+质量20%+技术沉淀15%+其他5%。这个框架运行了超过一年，经历了三次迭代：
>
> 第一阶段是规则确立，所有指标量化、数据源明确（禅道系统、测试组输出），减少主观争议；
>
> 第二阶段发现固定分值导致刷分问题，引入弹性封顶（比如文档从固定+2分改为+1分、3分封顶），从数量导向转为质量导向；
>
> 第三阶段意识到纯量化指标有盲区，增加了主管主观判断权（月度测试质量±5分、CodeView代码质量-3到3分），同时引入探索型任务认可研究性工作。
>
> 执行效果：团队质量基线被内化（Bug三项指标大多数人能拿满分），分数区分度适中（70~80分区间），半年汰换50%带动团队成长。"

### 5.2 体现的管理能力

| 能力 | 证据 |
|------|------|
| 数据驱动管理 | 所有指标有明确数据源，工时计算规则精确 |
| 机制设计能力 | 权重骨架稳定，细则持续迭代，形成PDCA闭环 |
| 问题发现与解决 | 发现刷分→引入封顶；发现量化盲区→引入主观判断 |
| 团队建设 | 质量基线内化、技术沉淀制度化、探索型任务鼓励创新 |
| 管理成熟度进阶 | 从L1规则驱动→L2机制驱动→L3文化驱动 |
