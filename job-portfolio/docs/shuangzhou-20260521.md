## 一、双周迭代的完整画像

### 1.1 什么是"双周迭代"

**周期**：2周（约9~10个有效工作日）为一个迭代周期。

**错位1周的三岗流水线**：

```
Week 1:
  产品：输出需求 → 需求评审 → 确定负责人 → 开发反讲
  研发：上一迭代收尾（联调/测试/上线）
  测试：上一迭代验收 / 本迭代用例准备

Week 2:
  产品：需求变更处理 / 下一迭代准备
  研发：本迭代开发 → 联调 → 提测
  测试：本迭代测试 → BUG回归 → 验收
```

**核心设计**：产品、研发、测试三个岗位的工作流**错开1周**，避免"需求来了研发还没空、研发做完了测试还没准备好"的串联等待。

### 1.2 与项目负责制的关系

| 维度 | 项目负责制 | 双周迭代 |
|------|-----------|----------|
| **管理对象** | 单个项目（1人周以上的复杂需求） | 整个研发团队的所有需求 |
| **时间跨度** | 项目全生命周期 | 固定2周周期 |
| **责任人** | 项目负责人（开发轮流担任） | 各小组主管 |
| **关注重点** | 跨组协调、里程碑推进、质量把控 | 工时规划、负载均衡、交付节奏 |
| **关系** | 大项目嵌套在迭代内运行 | 迭代为大项目提供时间盒和资源约束 |

**一句话总结**：双周迭代是"节拍器"，确保团队稳定输出；项目负责制是"推进器"，确保复杂项目不跑偏。两者叠加形成**节奏+质量**的双重保障。

## 二、六大管理工具详解

### 2.1 工具一：业务部门双周例会 → 需求收集表

**文件**：`00-原始需求收集表.xlsx`（国内）+ `00-国际需求收集表.xlsx`（国际）

**流程**：
```
会前：业务人员按模板填写"原始问题收集表"
       ↓
会上：产品评估需求价值、排优先级
       ↓
会后：确认转研发的需求录入"需求跟进表"
```

**原始问题收集表字段**：
| 字段 | 说明 |
|------|------|
| 问题编号 | P开头流水号（如P0722） |
| 一句话描述 | "作为xxx角色，在xxx场景下，遇到xxx问题" |
| 提出人/填写人 | 业务方 |
| 问题分类 | 人效低 / 客户服务差 / 可以增加潜在收入 / 存在风险 / 其他 |
| 影响业务范围 | 3D / CNC / 国际 / 国内官网/后台 等 |
| 发生频次 | 量化数据 |
| 潜在收入估计 | 如果解决能带来的收益 |

**需求跟进表字段**：
| 字段 | 说明 |
|------|------|
| 问题编号→需求编号 | Pxxxx → Rxxxx |
| 需求名称 | 标准化命名 |
| 需求评估分数 | 70分/46分等（优先级量化） |
| 工作量预估 | 1人日以下 / 1~5人日 / 5~10人日 / 10人日以上 |
| 所属大模块 | 国内官网/后台 / 国际后台 / CRM 等 |
| Owner | 产品负责人 |
| 需求进度反馈 | 待开发 / 开发中 / 已上线 / 暂时挂起 / 无需处理 |
| 禅道编号 | 关联禅道系统 |
| 上线时间 | 实际发布时间 |

**关键发现**：需求从提出到上线有完整的**编号追踪链**（P问题编号 → R需求编号 → 禅道编号），每个需求的状态一目了然。

### 2.2 工具二：产品需求录入禅道

**系统**：禅道项目管理工具

**使用方式**：
- 每个确认转研发的需求在禅道创建对应条目
- 需求拆分为任务，任务分配工时
- 开发人员领取任务并更新状态
- 所有工时数据、任务状态、Bug数据均来自禅道

**禅道在双周迭代中的角色**：
- 需求池管理
- 任务拆分与分配
- 工时录入与统计
- Bug跟踪与管理
- 迭代进度可视化

### 2.3 工具三：研发双周例会 → 迭代周期表

**文件**：`01-Java迭代周期.xlsx` / `01-OMS迭代周期.xlsx`

**核心参数**：

| 参数 | Java组 | OMS组 | 前端组 |
|------|--------|-------|--------|
| 时间盒容量 | 0.7 | 0.7 | 0.8 |
| 个人基础工时 | 44.1h | 44.1h | 50.4h |

**计算公式**：
```
时间盒 = 有效工作日 × 7工时/天 × 时间盒容量
个人基础工时 = 时间盒 / 迭代人数
可用工时 = 个人基础工时 - 预留工时
负载率 = 实际领取工时 / 个人基础工时
```

**迭代周期表结构**：

| 字段 | 说明 |
|------|------|
| 迭代开始/结束日期 | 固定2周 |
| 有效工作日 | 9~10天 |
| 迭代人数 | 小组人数 |
| 开发 | 成员姓名 |
| 个人基础工时 | 时间盒/人数 |
| 指派任务工时 | 已分配的具体任务工时 |
| 预留工时 | 未明确分配但预留的工时 |
| 本迭代剩余工时 | 基础工时 - 指派工时 |
| 未指派任务工时 | 待分配的需求工时 |
| 可用工时 | 可再分配的工时 |

**例会产出**：
1. 统计迭代完成负载数据
2. 整理迭代需求完成进度
3. 统计下个迭代预留工时（区分评估工时、预留工时）
4. 内部优化项目提前沟通

**发布后问题记录**：单独sheet记录上线后发现的问题（问题描述、需求、代码定位、是否已解决）

### 2.4 工具四：双周迭代过程管理 → 负载情况盘点

**文件**：`04-业务中心负载情况盘点（过程数据）.xlsx`

**统计频率**：每周五17:30（期初）+ 迭代中复盘（期末）

**盘点维度**：

| 维度 | 说明 |
|------|------|
| 计划工时 | 有效工作日 × 7 × 70%（与绩效公式一致） |
| 实际领取 | 成员在禅道领取的任务工时 |
| 累计任务数/工时 | 迭代累计数据 |
| 负载率 | 实际 / 计划 |
| 存量储备 | 未拆解工时的任务列表 |
| 必要说明 | 每个人的状态说明（休假、待评审、待反讲等） |

**负载率计算公式**：
```
计划 = 有效工作日 × 7 × 70%
负载率 = 实际领取工时 / 计划
```

**示例数据**（Java组 260508~260521迭代）：
| 人员 | 计划 | 实际(期初) | 负载率(期初) | 实际(期末) | 负载率(期末) |
|------|------|-----------|-------------|-----------|-------------|
| 露全 | 49 | 5任务/50h | 0.71 | 5任务/50h | 0.71 |
| 世琦 | 49 | 8任务/40h | 0.57 | 9任务/55h | 0.79 |

**演进轨迹**：
- 早期（2406~2408）：简单版（计划/实际/负载率）
- 中期（2409~2412）：增加指派/完成/环比
- 后期（2501~2605）：增加完成率、存量储备管理、必要说明

### 2.6 工具六：在线问题管理 → 线上问题记录表

**文件**：`2604~线上问题记录表.xlsx`

**字段结构**：

| 字段 | 说明 |
|------|------|
| 编号 | 2604001、2605001等（年月+流水） |
| 问题描述 | 详细描述 |
| 系统 | OMS / MES / SCM / JAVA |
| 严重程度 | P1致命 / P2严重 / P3一般 / P4建议 |
| 录入日期 | 发现问题日期 |
| 录入人 | 记录人 |
| 发现问题人 | 谁发现的 |
| 问题类型 | 系统bug / 需求缺陷 / 设计如此 / 外部原因 / 配置问题 |
| 原因&解决方案 | 根因分析+修复方案 |
| 问题状态 | 已解决 / 转需求 / 挂起 / 无需处理 |

**Bug严重程度定义**：
- P1：致命（系统崩溃、死机、死循环、数据丢失）
- P2：严重（主要功能丧失、数据丢失、一级菜单不可用）
- P3：一般（功能未完全实现但不影响使用）
- P4：建议（界面、性能缺陷，不影响操作）

**处理时效**：
- P1/P2：日清（立即处理）
- P3：正常进度处理
- P4：可视情况延后

### 2.7 工具七：工时评估机制 → 前端工时预估

**文件**：`前端工时预估-苏坚生--工时评估机制.xlsx`

**统一口径**（2026年3月11日起执行）：

1. **迭代会按现场出牌**（只在禅道录入）
2. **紧急需求且<7工时**：可当场先拍，事后再修正（评估人1登记拍脑袋，对外工时登记最终工时）
3. **其他全部JS+2前端评估后再输出**：任何需求都登记，前端尽可能保持2人，JS全部评估，协同一致之后对外发布

**评估原则**：
- 按功能和接口拆分评估
- 工时评估**不含联调**，按30%比例给联调buff
- 全新页面考虑控件复用
- 评估依据：蓝湖原型 + 禅道任务

**评估模板字段**：

| 字段 | 说明 |
|------|------|
| 系统 | OMS / MES / 官网 / 小程序 |
| 模块/页面 | 具体页面 |
| 任务名称 | 功能点 |
| 说明（评估依据） | 详细的功能拆解和工时依据 |
| 工时(H)-jedsu | 苏坚生评估 |
| 工时（美琴/暖钦/建奇） | 前端各自评估 |

**多人评估取一致**：评估人1 + 评估人2，协同一致后对外发布，禅道录入对外发布的工时明细。

## 三、绩效数据链路 — 从管理工具到考核分数

这是整个体系最关键的发现：**所有绩效指标都有明确的数据来源，不是主观打分。**

### 3.2 各绩效指标的数据来源对照

| 绩效指标 | 标准分 | 数据来源文件 | 具体数据字段 |
|----------|--------|-------------|-------------|
| **有效工时评分** | 40分 | `04-业务中心负载情况盘点` | 负载率 = 实际领取工时 / (有效工作日×7×70%) |
| **交付及时性** | 20分 | `02-需要测试介入的需求池` | 计划提测/上线时间 vs 实际提测/上线时间 |
| **Bug修复及时率** | 10分 | 禅道系统 | Bug创建时间 vs 修复时间，按级别计算延迟天数 |
| **二次Bug率** | 5分 | 禅道系统 | 同一Bug回归次数 / 项目人天 |
| **当月遗留缺陷** | 5分 | 禅道系统 | 当月创建且当月未关闭的Bug数量 |
| **技术沉淀** | 15分 | 各小组主管汇总 | 设计文档/培训/手册/反讲/项目跟进人加分 |
| **其他** | 5分 | 考勤记录+主管汇总 | 每日同步/禅道维护/迟到早退/客户投诉 |

### 3.3 数据驱动的核心设计

**为什么这个链路是有效的？**

1. **去主观化**：绩效分数不是主管拍脑袋，而是从日常管理工具中自动提取
2. **实时可追溯**：每个分数都能追溯到具体的禅道任务、需求池记录、负载盘点
3. **减少争议**：开发人员可以自查自己的负载率、提测延期记录、Bug修复时效
4. **管理即考核**：日常管理工具本身就是考核数据源，不增加额外管理成本

## 五、面试价值提炼

### 5.1 可复用的话术

**关于双周迭代体系：**

> "我搭建了一套双周迭代的研发管理体系，核心是'节拍器+数据驱动'。节拍器是指产品→研发→测试错位1周的双周流水线，确保团队稳定输出节奏；数据驱动是指所有绩效指标都有明确的数据来源——有效工时来自负载盘点、交付及时性来自测试需求池、质量指标来自禅道Bug数据。
>
> 具体做法上，我们有7个管理工具环环相扣：业务部门双周例会产生需求 → 产品录入禅道 → 研发双周例会做任务分配 → 每周五做负载盘点 → 测试跟踪提测上线时间 → 上线后记录线上问题 → 前端有独立的工时评估机制（2人评估一致后对外发布）。
>
> 这套体系运行下来，绩效不是主管拍脑袋，开发人员可以自查自己的负载率和延期记录，争议大幅减少。团队负载率稳定在70%~90%区间，质量基线内化后Bug三项指标多数人能拿满分。"

### 5.2 体现的管理能力
