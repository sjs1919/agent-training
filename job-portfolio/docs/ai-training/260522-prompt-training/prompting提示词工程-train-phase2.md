## 1. 开场：作业点评 + 问题收集（5 min）

### 1.1 作业点评（3 min）

展示 2-3 个优秀作业（改造前 vs 改造后的 prompt），指出亮点：

- 具体约束是否到位
- 模板参数化是否可复用
- 是否建立了个人模板库

### 1.2 问题收集（2 min）

提前收集学员在第 1 期后实践中遇到的真实问题，分类贴到白板上：

| 类别 | 典型问题 |
|------|---------|
| **上下文丢失** | 对话长了 AI 就忘了之前的约定 |
| **多文件改动** | 改了一半发现方向错了，怎么回退 |
| **代码库太大** | AI 找不到该读哪个文件 |
| **AI 编造 API** | 生成了不存在的方法或库 |
| **测试覆盖不足** | AI 写的测试只有 Happy Path |

> 这些问题就是今天课程要解决的。

## 2. 多轮对话状态管理（12 min）

### 2.1 核心认知：对话不是无限的（2 min）

Claude Code 的上下文窗口约 200K tokens，但：

- 对话越长，早期消息的权重越低
- 超过阈值后，早期内容会被丢弃
- 症状：AI "变笨"、忘了之前的约定、重复已做过的事

### 2.2 断点续传技巧（4 min）

**问题**：一个功能开发到一半，对话上下文已经混乱，怎么办？

**传统做法**（不推荐）：
```
继续刚才的功能
```
→ AI 大概率忘了做到哪一步，需要反复确认。

**正确做法**：手动总结进度 + 开新对话

```
我们刚才完成了以下工作：
1. 在 src/api/order.ts 中创建了 createOrder 接口
2. 在 src/services/order.ts 中实现了核心逻辑
3. 待完成：联调前端表单验证 + 写单元测试

当前状态：后端接口已通，返回格式为 {code, data, message}

请继续第 3 项：在 src/views/order/ 下创建下单表单页面，
要求：
- 表单字段：product_id(下拉选择), quantity(数字), remark(文本)
- 提交后调 POST /api/order/create
- 用现有的 RePureTableBar + vxe-table 风格
```

**进阶技巧**：用文件保存进度状态

创建一个 `tmp/ai-progress.md`：
```markdown
# AI 协作进度追踪

## 当前任务
在 lexyAdmin 中实现订单管理模块

## 已完成
- [x] API 接口：src/api/order.ts
- [x] 服务层：src/services/order.ts

## 待办
- [ ] 前端表单页面
- [ ] 单元测试
- [ ] 权限控制

## 关键约定
- 返回格式统一：{code: 0, data, message}
- 表格用 RePureTableBar + vxe-table
- 按钮权限用 RePerms 组件
```

然后告诉 AI：
```
读 tmp/ai-progress.md，了解当前进度和约定，然后继续待办列表中的第 1 项。
```

### 2.3 `/clear` 的正确用法（2 min）

| 场景 | 做法 |
|------|------|
| 对话上下文混乱，AI 表现异常 | `/clear` 开新对话，带上项目配置文件 |
| 完成一个大任务，开始新任务 | `/clear` + 手动总结进度到文件 |
| 发现 AI 在重复之前的错误 | `/clear` 后，把之前的教训写进新 prompt |

**关键**：`/clear` 后，重要的项目约定不会丢失（因为 `CLAUDE.md` 会自动加载），但**当前进度会丢失**，必须手动保存。

### 2.4 用文件传递上下文（4 min）

**反模式**：把大段代码贴到对话框

```
❌ 帮我看下这段代码有什么问题：
[paste 200 行代码]
```
→ 浪费上下文窗口，AI 只能看到片段。

**正确做法**：让 AI 读文件

```
✅ 读 src/services/order.ts 中的 createOrder 函数，
分析是否有并发安全问题。
重点关注：
1. 库存扣减逻辑是否原子
2. 重复订单的判断机制
3. 事务边界
```

**文件传递 vs 对话框粘贴的对比**：

| 方式 | 适用场景 | 优点 |
|------|---------|------|
| 对话框粘贴 | 小于 50 行的片段 | 快速，AI 能直接看到 |
| 让 AI 读文件 | 超过 50 行、跨文件 | 不占用上下文窗口，AI 可以反复查阅 |
| 写进临时文件 | 进度追踪、复杂约定 | 持久化，断点续传 |

## 3. 大型代码库导航策略（10 min）

### 3.1 先建项目地图（3 min）

**问题**：进入一个陌生的大型项目，不知道从哪开始。

**传统做法**：自己花几小时读目录结构、看入口文件。

**AI 辅助做法**：让 AI 先分析项目结构

```
先分析这个项目（当前目录）的整体结构：
1. 技术栈和框架
2. 目录组织方式
3. 核心业务模块有哪些
4. 入口文件和关键配置

用 markdown 列表输出，不要改任何代码。
```

**进阶**：让 AI 生成项目地图文件

```
基于你对项目的分析，生成一个 PROJECT_MAP.md，包含：
- 模块清单和职责
- 关键文件位置
- 数据流向
- 技术决策记录

写到 docs/PROJECT_MAP.md，后续开发时我会让你参考它。
```

### 3.2 跨模块依赖分析（3 min）

**场景**：要改 `src/services/order.ts`，怎么知道会影响哪些模块？

```
分析 src/services/order.ts 被哪些文件引用，
列出直接调用方和间接调用方。

如果我要修改 createOrder 的返回格式，
评估影响范围，列出需要同步修改的文件清单。
```

**关键价值**：AI 能快速做静态分析，人只需要评估业务影响。

### 3.3 精确引用 vs 全量投喂（4 min）

**反模式**：让 AI "看整个 src/ 目录"

```
❌ 看看 src/ 下的代码，给我优化建议
```
→ AI 注意力分散，给出的建议泛泛而谈。

**正确做法**：精确引用相关文件

```
✅ 读以下文件，分析订单模块的性能瓶颈：
- src/services/order.ts（核心逻辑）
- src/models/order.ts（数据模型）
- src/api/order.ts（接口层）

重点关注：
1. 是否有 N+1 查询
2. 循环中是否有 DB 查询
3. 大对象是否全量加载
```

**原则**：一次给 AI 的文件不超过 5-10 个，总代码量不超过 500 行。

## 4. AI 辅助 TDD（10 min）

### 4.1 为什么需要 AI 辅助 TDD（2 min）

传统 TDD 的痛点：
- 写测试耗时，容易遗漏边界
- 人倾向于写 Happy Path 测试
- 需求变了，测试同步更新成本高

AI 能做什么：
- 根据需求自动生成测试用例（含边界）
- 根据失败测试生成实现
- 需求变更时自动建议测试更新

### 4.2 AI 辅助 TDD 流程（4 min）

**标准流程**：

```
第 1 步：把需求描述给 AI，让它生成测试

"我要实现一个订单折扣计算函数，规则如下：
- 订单金额 >= 1000，打 9 折
- 订单金额 >= 5000，打 8 折
- VIP 用户额外 95 折（可与上面叠加）
- 金额必须 > 0

请为 calculateDiscount(orderAmount, isVip) 生成完整的单元测试，
用 Vitest，覆盖所有分支和边界情况。"

第 2 步：Review AI 生成的测试，确认边界覆盖

第 3 步：让 AI 根据测试写实现

"测试已通过，现在根据这些测试用例，
实现 calculateDiscount 函数。
要求：
- 用 TypeScript
- 参数校验：orderAmount 必须 > 0，否则抛错
- 逻辑清晰，不要过度设计"

第 4 步：跑测试，红 → 绿 → 重构

第 5 步：让 AI 检查测试覆盖率

"用 Vitest 的 coverage 模式跑测试，
如果覆盖率低于 80%，补充缺失的测试用例。"
```

**关键原则**：AI 生成，人 Review。不要无脑接受 AI 的测试和实现。

### 4.3 Prompt AI 覆盖边界情况（2 min）

**常见问题**：AI 写的测试只有正常路径。

**解决方案**：在 prompt 中明确要求边界覆盖

```
生成单元测试时，必须包含以下边界情况：
1. 最小值 / 最大值
2. 空值 / null / undefined
3. 类型错误输入
4. 并发场景（如果适用）
5. 特殊字符 / 超长输入

用 describe.each 或 it.each 组织同类边界测试。
```

### 4.4 测试作为安全网（2 min）

**核心认知**：AI 生成的代码可能出错，测试是最后一道防线。

**工作流**：
1. 让 AI 改代码前，先确保现有测试能通过
2. AI 改完后，跑全量测试
3. 测试通过后才允许 Review

```
先跑 npm run test:unit，确认所有现有测试通过。
然后修改 src/services/order.ts 中的 createOrder 函数。
修改完成后，再次跑测试，确保没有破坏现有功能。
如果有失败的测试，优先修复，不要跳过。
```

## 5. Claude Code 深度工作流（8 min）

### 5.1 CLAUDE.md 进阶写法（3 min）

**基础版**（第 1 期已讲）：项目技术栈、代码风格、常用命令

**进阶版**：加入团队协作规范

```markdown
# CLAUDE.md — lexyAdmin 项目规范

## 技术栈
Vue 3.4 + TypeScript + Vite + Pinia + Element Plus

## 代码规范
- 命名：组件 PascalCase，函数 camelCase，常量 UPPER_SNAKE_CASE
- 类型：所有 API 入参出参必须定义 interface，放在 types/ 目录
- 注释：复杂逻辑用中文注释，说明"为什么"而非"做什么"

## 架构约定
- API 层：src/api/，统一用 http.request<T>(method, url, config)
- Store 层：src/store/modules/，用 defineStore，export useXxxStoreHook
- 视图层：src/views/，按模块分目录，组件放在对应 views/ 下的 components/

## 禁止事项
- 不要在组件里直接调 API，必须走 store action
- 不要用 any，类型不确定时用 unknown + 类型守卫
- 不要引入新的 UI 库，现有 Element Plus 已覆盖需求

## 测试要求
- 框架：Vitest
- 覆盖率：statements/branches/functions/lines >= 80%
- 新功能必须包含单元测试

## 常用命令
- npm run dev：启动开发服务器
- npm run test:unit：跑单元测试
- npm run typecheck：TypeScript 类型检查
- npm run precommit：提交前检查
```

**关键**：CLAUDE.md 不是一次性写完的，是在项目演进中不断追加的。

### 5.2 settings.json 实战（3 min）

Claude Code 的配置文件 `.claude/settings.json`：

```json
{
  "permissions": {
    "allow": [
      {
        "command": "npm run test:unit",
        "description": "运行单元测试"
      },
      {
        "command": "npm run typecheck",
        "description": "TypeScript 类型检查"
      }
    ]
  },
  "hooks": {
    "pre-commit": "npm run precommit"
  }
}
```

**作用**：
- `permissions.allow`：预先授权常用命令，减少每次的权限提示
- `hooks.pre-commit`：提交前自动跑检查，防止低级错误

### 5.3 高级命令速查（2 min）

| 命令 | 用途 | 场景 |
|------|------|------|
| `/ask` | 只提问，不改代码 | 想先理解代码再决定怎么改 |
| `/test` | 运行测试 | 改完代码快速验证 |
| `/commit` | 自动提交 | AI 完成一个完整功能后 |
| `/clear` | 清空对话上下文 | AI 变笨了，或开始新任务 |
| `/undo` | 撤销最后一次改动 | AI 改错了，快速回退 |

## 6. 完整功能开发演练（12 min）

### 演练目标

带一个真实需求走完整流程：**在 lexyAdmin 中实现"订单管理"模块**

### 流程演示

**第 1 步：需求分析（让 AI 读需求，给出方案）**

```
需求：在 lexyAdmin 中增加订单管理功能，包含：
1. 订单列表页（订单号、客户名、金额、状态、创建时间）
2. 支持按状态筛选、按时间范围筛选
3. 支持导出 Excel
4. 订单详情弹窗

先分析需要改哪些文件，列出清单让我确认。
不要直接改代码。
```

**第 2 步：确认方案后，生成测试（TDD 第一步）**

```
方案已确认。现在为订单模块生成单元测试：
- 测试 src/services/order.ts 中的核心逻辑
- 覆盖：创建订单、状态流转、金额计算
- 边界：金额为 0、负数、超大数

用 Vitest，写在 tests/unit/services/order.test.ts。
```

**第 3 步：写实现**

```
测试已通过。现在实现订单列表页：
- 位置：src/views/order/index.vue
- 参考：src/views/system/user/index.vue 的结构
- 表格：RePureTableBar + vxe-table
- 筛选：顶部搜索栏，按状态和日期范围
- API：src/api/order.ts，参照 src/api/user.ts 的写法
```

**第 4 步：验证 + Review**

```
跑 npm run test:unit 和 npm run typecheck，
确保没有破坏现有功能。
然后 Review 生成的代码，检查：
1. 命名是否符合项目规范
2. 是否有重复代码可以提取
3. 错误处理是否完善
```

**第 5 步：保存进度**

```
把今天的进度更新到 tmp/ai-progress.md：
- 已完成：订单列表页 + 单元测试
- 待办：订单详情弹窗 + Excel 导出
- 关键决策：表格复用了 user 页面的 RePureTableBar 配置
```

## 附录 A：讲师备忘 — 常见问题应对

### Q1：学员说 "AI 生成的测试质量不高"

**回应**：测试质量取决于 prompt 的质量。给 AI 明确的边界要求（"必须覆盖 null/空数组/超大数"），质量会显著提升。另外，AI 是生成初稿，人必须 Review 和补充。

### Q2：学员说 "我们的项目没有测试，怎么开始 TDD"

**回应**：不是从零开始 TDD。可以先用 AI 给**现有代码**补测试（作为安全网），然后再用 TDD 开发新功能。

### Q3：学员说 "CLAUDE.md 写了很多，AI 还是记不住"

**回应**：检查两点：
1. CLAUDE.md 是否在项目根目录
2. 内容是否过长（超过 500 行可能超出上下文）

解决方案：拆分为多个文件，`CLAUDE.md` 放核心约定，具体模块的规范放在 `docs/standards/` 下，需要时让 AI 读。

### Q4：学员说 "AI 改代码把别人的功能弄坏了"

**回应**：这是没有跑全量测试的后果。养成习惯：**改前跑测试 → 改后跑测试 → 测试通过才提交**。用 `settings.json` 的 `pre-commit` hook 强制执行。
