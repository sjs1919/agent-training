> 本文档以 `主负责人.xmind` 思维导图为基准，覆盖项目负责人权责机制与全流程管理方法。

## 开宗立意

> **项目不跟，等于废弃。**
> **多人负责，等于没人负责！**

主负责人的价值不在于写多少代码，而在于 **让正确的事情在正确的时间发生**。

## 一、权责相匹（5 分钟）

### 核心理念

> 只有复杂需求或项目才需要项目负责人。
>
> 干得多、管得宽 → 有激励；
> 出了问题、延期了 → 有责任。

### 奖励

| 评分 | 标准 | 说明 |
|------|------|------|
| 1 分 | 表现一般，按时完成项目 | 基础完成分 |
| 2 分 | 表现良好，项目质量高 | 主管评定 |
| 3 分 | 表现优秀，超预期交付 | 主管评定，**3 分封顶** |

- 以**完成**为标准，积分与绩效挂钩。
- **1 分 = 1 天 = 7 工时**。

### 惩罚

| 扣分 | 标准 | 说明 |
|------|------|------|
| 1 分 | 一般失误，影响项目进度或质量 | 如延期提测、未统一协调等 |
| 2 分 | 较大失误，造成明显影响 | 如联调失控、分别提测导致测试返工等 |
| 3 分 | **重大失误**，造成严重后果 | 如上线后严重故障、未做 double check 导致事故等 |

- 项目负责人承担主责，项目组成员 **减半**（如项目负责人扣 2 分，成员扣 1 分）。
- 加分应在项目上线所在的月份。
- 超过1分的加分，由主管根据项目的困难程度和完成的质量做评估。
- 绩效排行（直接影响效果）：eg：榜首 4 星（1.2 倍工资），榜尾 2 星（0.8 倍工资）。

**未做项目复盘：扣 1 分。**

**扣分示例**：某项目负责人未组织项目复盘，扣 1 分；同时因分别提测导致测试返工，扣 2 分。合计扣 3 分。项目组成员减半扣 1.5 分。

**正面案例**：某中等需求涉及 OMS + 前端两组，项目负责人老李提前两天协调各组自测，按时统一提测，测试一次通过无冒烟不通过，最终顺利上线。老李获得 1 分奖励，月度积分排名上升。

**反面案例**：某复杂需求没有明确的项目负责人，各组各自为战。OMS 组认为前端会适配他们的接口格式，前端则按老版本文档开发。提测当天才发现接口对不上，延期 5 天。如果有负责人提前组织对齐，本可避免。

### 需求变更免责

需求变更导致延期 → 上报主管+产品 → 测试变更里程碑 → 提供变更留痕证据 → 调整计划不计惩罚。

**原则：没有留痕的变更 = 开发担责。** 产品口头加的临时需求，没有任何 IM/邮件记录，延期后不能作为免责依据。

### 新手保护期

> 首次担任项目负责人的前 **2 个项目**，只记录积分，不执行扣分。

**为什么设置保护期？** 如果新手第一次当负责人就被扣 3.5 分，下个月工资直接从 10,000 降到 8,000，他这辈子都不会再接负责人了。保护期让新手在无风险环境中积累经验，第 3 个项目起正常执行奖惩。

### 一页纸总结

```
┌──────────────────────────────────────────┐
│  奖励：1-3 分（表现一般1分/良好2分/优秀3分）│
│  惩罚：1-3 分（一般1/较大2/重大3），成员减半 │
│  未复盘：扣 1 分                            │
│  连带责任：项目负责人主责 + 成员减半          │
│  免责：需求变更需留痕                       │
│  保护：前 2 个项目只记积分不扣分            │
└──────────────────────────────────────────┘
```

## 二、需求评审与开发反讲（8 分钟）

### 2.1 确定主负责人

**唯一标准：主流程关键路径所在组**

| 项目类型 | 建议人选 |
|---------|---------|
| 紧急 | **熟手** |
| 非紧急 | **人才池轮流**（绩效绩优 / 有潜力新人 + 有意愿者入池） |

- **t+1 天**确定，人不在可等，太久主管拍板。
- 同步确定反讲时间。

### 2.2 开发反讲

> **1 人周以上的项目，必须有开发反讲。**

**参与方**：产品 + 开发 + 测试（全部参加）

**主要目的**：对齐开发思路，对齐项目信息，最终敲定开发方案

**角色分工**：
- 开发主讲方案（项目成员代讲）
- 产品 + 开发成员 + 测试挑毛病，补齐项目信息和开发方案
- **全员发言，避免开发一言堂**

**输出**：任务分解 + 工时评估 + 联调时间约定

**工时评估**：主管组织统一评估，不一致时全组评估取合理值，以关键人最长路径为项目周期。

**正面案例**：某复杂需求涉及 OMS + MES + 前端三组，反讲会上 OMS 组项目成员小张详细讲解了订单推送到生产线的数据流转方案，前端小李发现回调接口少了一个"状态码"字段，当场补齐。测试小赵提出边界场景"订单取消时 MES 如何处理"，开发确认后补充了用例。反讲后各组接口文档统一更新，联调 2 天完成。

**反面案例**：某 1.5 人周的需求跳过了反讲，OMS 组按自己的节奏开发，前端按产品原型图做页面。提测后才发现前端展示的是"订单金额"，OMS 传的是"订单数量"，字段含义根本没对齐。返工花了 4 天，项目延期一周。如果做了反讲，这个问题 5 分钟就能暴露。

### 一页纸总结

```
┌──────────────────────────────────────────┐
│  需求评审 → t+1 确定主负责人               │
│           → 同步确定反讲时间               │
│                                          │
│  反讲（1人周+必须做）                      │
│  ├─ 全员参加，项目成员代讲                 │
│  └─ 输出：任务分解 + 工时评估 + 联调时间    │
└──────────────────────────────────────────┘
```

## 三、开发阶段（8 分钟）

### 3.1 关注接口提前对齐

- 前后端/系统间接口文档应在开发初期输出，而非联调前才临时对接
- 项目负责人牵头组织接口评审，确认字段、格式、调用时机
- 接口对齐后，各组可并行开发，减少相互等待

**正面案例**：某项目涉及 OMS + 前端 + MES 三组，项目负责人小王在开发第 2 天就组织了三方接口评审，确认了 12 个接口的字段定义和返回格式。评审后各组并行开发，前端用 Mock 数据先写页面，OMS 和 MES 各自实现接口逻辑。联调时只发现 1 个字段拼写错误，2 小时修复，整体进度比预期还快了半天。

**反面案例**：某项目没有提前对齐接口，前端按产品原型图自己猜字段名写页面，OMS 按数据库字段名输出接口。联调当天发现前端期望的 `orderAmount` 和 OMS 实际的 `amount` 对不上，`status` 字段前端以为返回 1/2/3，OMS 实际返回的是 `pending`/`paid`/`shipped`。光是字段对齐就花了 2 天，联调从 3 天拖到 5 天，项目延期上线。

### 3.2 进度会议

| 维度 | 要求 |
|------|------|
| 频次 | 每周 **2 次**（周三、周五） |
| 范围 | 按项目虚拟团队（约 5 人） |

**三段式**：会前（成员自行翻转禅道状态）→ 会中（看板+燃尽图参考+问题清单）→ 会后（主负责人推动解决）

### 3.3 关注联调时间

- 联调是最容易失控的环节
- 日常私下跟进，仍耽误则会上公开追问
- 提前确定联调里程碑，暴露接口依赖

**正面案例**：项目负责人老王在周一下午就私下问了各组接口准备情况，发现 MES 组的回调接口还没输出。老王当场协调 MES 主管加急，周三联调按时开始，周五顺利完成。

**反面案例**：某项目联调 deadline 是周四，项目负责人小张周二才发现前端和 OMS 的接口字段还没对齐。OMS 组说"再给我们两天"，前端说"那我们只能等"。联调从周四拖到下周二，整个迭代节奏被打乱。如果小张周一就跟进，本可以提前暴露问题。

### 3.4 关注测试用例评审

- 按项目优先级和提测时间排序，项目负责人之间协调
- 协调不了 → 上报风险 → 主管和总监定夺

### 一页纸总结

```
┌──────────────────────────────────────────┐
│  接口对齐：开发初期输出，联调前评审确认      │
│  进度会议：每周 2 次（周三摸底、周五对表）  │
│  联调：私下跟进 → 公开追问                 │
│  用例：按优先级排序，协调不了上报           │
└──────────────────────────────────────────┘
```

## 四、测试与上线（10 分钟）

### 4.1 测试阶段

**提测 —— 统一**

- 测试提供统一分支和打包版本
- 项目负责人协调统一时间提测，**不允许分别提测**

**为什么不允许分别提测？** A 组周一提测，B 组周三提测，测试拿到 A 组版本测了半天发现 B 组功能没进来，白测了。测试资源本就紧张（仅 2 人），反复无效测试浪费的是整个团队的时间。

**BUG 修复时效**

| 级别 | 定义 | 修复时效 | 示例 |
|------|------|---------|------|
| 1-2 级 | 阻塞/影响主流程 | **半天内** | 系统崩溃、核心功能不可用 |
| 3-4 级 | 非主流程/优化 | **24 小时内** | UI 显示错误、体验改进 |

### 4.2 上线发布

**上线时间排定**

- 初步排定：产品 + 项目负责人 + 成员 + 主管
- 最终敲定：测试分工会（根据实际资源和测试时间）
- 下发给项目负责人

**发布执行**

- 发布操作：运维执行
- 回滚决策：项目负责人决定

**版本回滚策略**

- 默认整包回滚
- 简单需求非常紧急：从测试分支拆分单独上线（或次日）

**发布后**：各组观察系统运行状况（1 小时 / 1 天 / 1 周）。

**正面案例**：某项目上线后，项目负责人小刘组织各组观察了 30 分钟，发现 OMS 服务 CPU 占用异常升高。当场回滚，排查后发现是一个查询未加索引。修复后重新发布，避免了一次线上故障。

**反面案例**：某项目上线后，各组以为"没问题"就下班了。第二天上午业务方反馈订单查询变慢，用户投诉。事后排查发现是发布时数据库连接池配置错误。如果上线后多观察 30 分钟，这个问题本可以第一时间发现。

### 一页纸总结

```
┌──────────────────────────────────────────┐
│  提测：统一分支、统一版本、统一时间        │
│  BUG：重要半天 / 次级 24h                  │
│  上线：测试分工会敲定 → 运维执行 → 项目决策回滚│
│  发布后观察：1h / 1d / 1w                  │
└──────────────────────────────────────────┘
```

## 五、项目复盘（5 分钟）

> 每个项目上线后，必须进行复盘。**未做复盘：项目负责人扣 1 分。**

### 5.1 复盘目的

> 复盘不是为了批判，而是为了**寻找更好的推动项目进行的方法论**。批评和自我批评是必要的，但**有度**。

- 总结项目中的亮光和不足
- 沉淀可复用的经验
- **绩效跟踪：记录项目负责人的积分和表现，作为人才池动态调整的依据**

### 5.2 复盘范围与流程

**复盘范围：仅复杂项目（1 人周+），每月约 1-3 个。**

**主持人：项目负责人。**

| 步骤 | 内容 | 时长 |
|------|------|------|
| 1 | 数据回顾（里程碑结果、问题与克服方法） | 5 min |
| 2 | 亮点总结（点名表扬优秀成员） | 5 min |
| 3 | 问题归因（匿名提出，不点名） | 10 min |
| 4 | 改进措施（聚焦提点最多的 1-2 个，重点讨论落地） | 5 min |
| 5 | 绩效记录（确认积分，公开评价） | 5 min |

**说明：问题环节聚焦于事情和问题，务必避免态度和人格**
**改进措施闭环**：下一个项目复盘时，简单检验上次改进措施的落实情况。

### 5.3 绩效跟踪

- 复盘时确认项目最终积分（奖励分 - 惩罚分）
- **绩效数据完全公开**：每月公开每个人的每个维度（为什么扣分、为什么奖励、什么项目拖累）
- 月度汇总后纳入绩效星级评定
- 表现优异者优先纳入下一轮人才池；表现不达标者暂停轮换资格
- **绩效反馈促进制度完善**：通过复盘数据持续优化项目负责人机制

**正面案例**：某项目复盘时，大家一致反馈"联调时间不确定"是最大痛点。团队决定增加"联调前 2 天确认接口文档"的硬性要求，并在下一个项目中试行。结果下一个项目联调从平均 4 天缩短到 2 天。

**反面案例**：某团队每月都做复盘，但每次复盘只记录积分、表扬几句，从不讨论改进措施。半年后复盘记录积累了 20 多份，团队遇到的问题还是那些老问题，复盘变成了"走过场"。

### 一页纸总结

```
┌──────────────────────────────────────────┐
│  项目复盘                                 │
│  ├─ 数据回顾（计划 vs 实际）               │
│  ├─ 亮点总结 + 问题归因                    │
│  ├─ 改进措施（1-2 个最关键）               │
│  └─ 绩效跟踪（积分确认 + 评价记录）         │
│                                          │
│  产出：复盘纪要 + 绩效更新                  │
└──────────────────────────────────────────┘
```

## 总结（3 分钟）

### 八句口诀

| 序号 | 口诀 | 含义 |
|------|------|------|
| 1 | **权责相匹** | 干得多有激励，延期了有责任，前期轻、后期重 |
| 2 | **关键路径** | 主负责人看关键路径，紧急用熟手、非紧急人才池轮流 |
| 3 | **反讲必做** | 1 人周以上必须有反讲，全员发言对齐方案 |
| 4 | **每周两回** | 周三摸底、周五对表，会前→会中→会后三段式 |
| 5 | **揪着不放** | 联调时间要"揪着各组负责人"推动 |
| 6 | **统一为要** | 统一提测、统一发布、统一顺序 |
| 7 | **double check** | 发布前最后确认（功能/配置/依赖/回滚/监控） |
| 8 | **手脚并用** | 组主管是项目负责人在各组的执行抓手 |
