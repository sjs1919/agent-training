## 二、行业面试要点与盲点（按技术领域最大化挖掘）

> 每个领域分三层：**要点**（必考）→ **盲点**（追问会倒的地方）→ **结合项目怎么答**

### 领域 1：Go 后端

**要点**
- Gin：路由分组、中间件链、Context 传值、参数绑定（ShouldBindJSON）
- GORM：模型关联（has many/belongs to）、事务（Transaction）、预加载（Preload）、软删除
- 分层架构：handler（参数校验+响应）→ service（业务编排）→ repository（CRUD）→ model
- 依赖注入：手工 DI vs wire vs fx
- 优雅关闭：信号监听、context 取消、超时控制
- 并发原语：goroutine、channel、sync.RWMutex、sync.WaitGroup、atomic
- 测试：表驱动测试、 testify mock、覆盖率

**盲点**
1. **Gin 中间件 Next()/Abort() 机制**：c.Next() 把控制权交给下一个中间件，c.Abort() 阻止后续中间件但不会 return 当前函数——很多人忘了手动 return
2. **GORM 事务嵌套**：嵌套的 `Transaction` 不是真嵌套事务，而是 SavePoint；真正隔离的嵌套事务需要不同 `*gorm.DB` 实例
3. **GORM N+1 查询**：Preload（多条 SQL）vs Joins（单条 SQL）vs Associations，性能差异大；不开 DB 日志容易漏
4. **优雅关闭时正在处理的请求**：`srv.Shutdown(ctx)` 会等正在处理的请求完成，但 SSE 长连接怎么办？需要 ctx 超时强杀
5. **goroutine 泄漏检测**：`go.uber.org/goleak` 或 pprof 的 goroutine profile；生产环境 goroutine 数量曲线是关键指标
6. **sync.RWMutex 写饥饿**：Go 1.9+ 引入饥饿模式，但读多写少场景仍可能延迟写锁；token-hub 密钥池用 RWMutex 要讲清这点
7. **context 泄漏**：`context.WithCancel` 必须调用 cancel()，否则 context 树泄漏；常见于提前 return 的错误路径
8. **map 并发读写**：`fatal error: concurrent map writes` 是不可 recover 的崩溃，不是 panic；必须用 sync.Map 或加锁
9. **interface 装箱与逃逸**：interface 参数会让值逃逸到堆；高频路径要避免
10. **Go 1.19 锁版本**（token-hub 真实）：面试官必问"为什么不用 1.21+？泛型（1.18+）为什么不用？升级路径？"——答：生产服务器限制，但这是技术债，需隔离运行环境或规划升级

**结合 token-hub 怎么答**
- 密钥池 RWMutex：讲读写分离收益，承认写饥饿风险，说"密钥池读远多于写，且写路径有 ReportFailure/TryRecover，频率可控"
- 手工 DI（`cmd/main.go`）：讲"项目规模中等，依赖图手工维护清晰，引入 wire/fx 增加心智负担不划算"
- 优雅关闭：讲"srv.Shutdown(ctx) 30s 超时 → 关 DB → 同步日志"的顺序，承认 SSE 长连接是未完全解决的点

### 领域 2：LLM API 网关 / 大模型应用工程

> 注：本领域只讲网关层（Key 管理、代理、计费），**不讲 Agent 开发**（未接触）

**要点**
- API Key 管理：多租户、配额、计费、sk-xxx 格式
- 密钥池：加权分配、故障熔断、自愈恢复、脱敏
- SSE 流式：Server-Sent Events 协议、流式转发、chunk 处理
- 多协议适配：OpenAI vs Claude（字段差异、流式结构差异）
- 计费：输入/输出/缓存 token、倍率模式、并发扣费

**盲点**
1. **SSE vs WebSocket vs HTTP/2 streaming**：为什么 LLM 用 SSE？SSE 单向（服务端→客户端）、文本、自动重连、基于 HTTP/1.1；WebSocket 双向但 LLM 场景不需要双向
2. **SSE 背压（backpressure）**：客户端处理慢，服务端 buffer 堆积怎么办？Go 的 `http.Flusher` + 检查写入错误；理论上需要流控
3. **SSE 断线重连**：`Last-Event-ID` 头；但 LLM 流式断开已生成的 token 怎么计费？token-hub 的做法是按已转发 chunk 计费
4. **Token 计数**：服务端计数（厂商返回 usage）vs 客户端计数（tiktoken）；缓存 token（prompt_cache_hit）怎么算？世纪互联 Claude 协议不返回缓存 token 是 token-hub 计费 BUG 的根因之一
5. **并发扣费原子性**（token-hub G04a P0）：Redis 原子操作（INCRBYFLOAT/DECRBY）+ Lua 脚本保证"扣费+写流水"原子；分布式锁性能差；预扣费（PreConsume）+ 结算（Settle）两阶段
6. **熔断恢复策略**：半开状态（half-open）放试探流量；token-hub 的 `TryRecover` 超过 retryInterval 标记候选恢复，但"候选"如何转"正式"有状态机
7. **多协议适配差异**：OpenAI 的 `tool_calls` 是数组、Claude 的 `tool_use` 是 content block；OpenAI 的 system 在 messages[0]、Claude 的 system 是顶层字段；流式 chunk 结构完全不同
8. **密钥脱敏**：日志 `maskKey`（保留前4后4），但前端展示、错误信息、监控面板都要脱敏；容易在某个路径漏掉
9. **失败请求不计费**：怎么界定"失败"？HTTP 5xx 算失败，但超时呢？流式中断呢？需要明确策略
10. **配额超限**：402 Payment Required 状态码；企业用户透支 vs 普通用户拒绝的策略差异；日消耗限额优先于透支

**结合 token-hub 怎么答**
- G04a 67/100：诚实讲"异常容错 45 分、可观测性 40 分、多模态计费 10 分"，重点讲 3 个 P0（并发竞态、企业池原子性、Redis 降级缺失）和改进方向
- 计费 BUG 亏损 97.7%：讲根因三连（世纪互联 Claude 不返回缓存 token + `calculator.go:49` 缓存双重计费 + ratio 配置错误），体现根因分析能力，重点是怎么发现的（对标 + 审计）
- 密钥池加权：讲 Pick() 累加权重 + 60000 次采样测试，承认加权分布的均匀性需要统计验证

### 领域 3：Vue 3 前端工程

**要点**
- Composition API：`<script setup>`、ref/reactive、computed、watch/watchEffect
- Pinia：setup store 风格、持久化、模块化
- Vite：构建、HMR、devServer 代理、分包
- Element Plus：组件库、主题定制、按需引入
- ECharts：K 线图、技术指标、大数据量性能
- TypeScript：strict 模式、类型推导、泛型

**盲点**
1. **ref vs reactive**：ref 要 `.value`（因为要支持基本类型 + 动态替换），reactive 不能整体替换；模板里 ref 自动解包
2. **响应式原理**：Vue3 用 Proxy（能监听新增/删除属性、数组索引），Vue2 用 Object.defineProperty（不能）；依赖收集在 getter，触发在 setter
3. **watch vs watchEffect**：watch 需显式指定依赖、惰性执行（除非 immediate），watchEffect 自动追踪依赖、立即执行；watch 能拿到新旧值
4. **computed 缓存**：依赖变化才重新计算；但依赖变化后值相同（如返回对象引用）会触发依赖它的 watcher，computed 本身不重算
5. **Pinia vs Vuex**：Pinia 去掉了 mutation、更好的 TS 推导、setup store 写法、模块化不需要 registerModule
6. **Vite 为什么快**：dev 用原生 ESM（按需编译）+ esbuild 预构建依赖（比 webpack 快 10-100 倍）；HMR 精确到模块边界
7. **Element Plus 按需引入**：unplugin-vue-components + unplugin-auto-import；CSS 体积从全量 1MB+ 降到按需
8. **ECharts 大数据量性能**（retrosys K 线）：几千根 K 线要用 `dataset` + `appendData` 增量、关闭动画、`large` 模式、web worker；这是 retrosys 架构开放问题之一（ECharts vs TradingVue vs Lightweight Charts）
9. **TypeScript strict 滥用 any**：strictNullChecks、noImplicitAny 开了但 `as any` 满天飞等于没开；要用类型守卫（is/in/typeof）替代断言
10. **defineProps/defineEmits 编译宏**：`<script setup>` 里不需要 import，编译期处理；运行时不存在

**结合 retrosys 怎么答**
- K 线库选型开放问题：讲"ECharts 通用但 K 线专业性弱、TradingVue 专注金融但生态小、Lightweight Charts 性能最好但 API 限制多"，MVP 倾向 ECharts，生产考虑 Lightweight Charts
- 双 Axios + 双通道认证：讲 web 和 admin 的隔离（不同 baseURL/拦截器/token 存储），webBearer 7天5设备 vs adminBearer 8h1设备的安全等级差异

### 领域 4：数据库设计

**要点**
- MySQL 8.0：InnoDB、utf8mb4、索引、事务隔离级别
- PostgreSQL：与 MySQL 差异、何时选 PG
- 迁移脚本：版本管理、回滚
- 引用完整性：外键 vs 应用层
- 索引：B+树、覆盖索引、最左前缀

**盲点**
1. **不用外键的代价**（token-hub 真实）：迁移灵活、性能好，但孤儿数据、一致性靠应用保证；面试官会追问"怎么保证？"——答：service 层事务 + 定时校验 + 监控孤儿数据
2. **事务隔离级别**：MySQL 默认 RR（可重复读），PG 默认 RC（读已提交）；RR 用 MVCC + 间隙锁防幻读，但有死锁风险；很多公司改 RC + 乐观锁
3. **utf8 vs utf8mb4**：MySQL 的 utf8 是 3 字节（存不了 emoji 和部分中文生僻字），utf8mb4 才是真 UTF-8；token-hub 用 utf8mb4 是对的
4. **索引失效场景**：函数操作（`WHERE DATE(create_time)=...`）、隐式类型转换（字符串列传数字）、OR 连接非索引列、LIKE `%xxx` 前缀通配
5. **联合索引最左前缀**：`(a,b,c)` 索引，`WHERE a=? AND b=?` 能用，`WHERE b=? AND c=?` 用不上；范围查询（`a > 1`）之后的列用不上索引
6. **PG vs MySQL 选型**（retrosys 双 DB）：PG 更适合复杂查询、JSONB、时序扩展（TimescaleDB）；MySQL 更成熟、运维生态好；retrosys 双 DB 的一致性是开放问题（X1）
7. **金融精度**（retrosys X5 矛盾）：必须用 Numeric/Decimal，绝不能用 Float；retrosys 的 database-guide 示例用了 Float 违反 coding-conventions 的 Numeric 硬性规则——这是文档内部矛盾，面试时讲"我们通过四角色评审发现并修正"
8. **迁移脚本回滚**：up/down migration，生产迁移要先备份 + 在 staging 验证 + 分批次；大表加索引要用 pt-online-schema-change 或 gh-ost
9. **分库分表**（token-hub 多租户）：按租户分库？还是共享库 + tenant_id？MVP 一般共享库，规模上来再分
10. **连接池配置**：GORM 的 SetMaxOpenConns/SetMaxIdleConns；retrosys 后端 MySQL pool=10/PG pool=20，数字怎么定要看 QPS 和连接成本

**结合项目怎么答**
- token-hub 不用外键：讲"迁移灵活 + 应用层保证"，诚实说代价（孤儿数据风险），讲补救（定时校验脚本）
- retrosys 双 DB：讲 MySQL（业务数据）+ PG（可能的时序/分析）分工，但承认跨库一致性是未解决问题

### 领域 5：工程化与 CI/CD

**要点**
- 规格驱动：spec→plan→task 三件套
- 文档体系：阶段编号、F/G 系列
- CI：GitHub Actions、job 并行、service container
- 质量门禁：覆盖率、lint、安全扫描
- CodeGraph：静态分析、影响范围

**盲点**
1. **规格驱动的成本**：文档维护成本高，何时该精简？答：MVP 阶段只写 spec + checklist，plan 在复杂模块才写；token-docs 文档量超代码量是过度
2. **CI 真实 DB vs mock**（retrosys）：真实 DB 能发现迁移/SQL 兼容问题，但慢、不稳定；service container 启动要 wait-for-db；mock 快但掩盖问题
3. **覆盖率 80% 门禁**：覆盖率不能保证质量（能测错的东西），但没有覆盖率会更糟；关键看分支覆盖而非行覆盖
4. **CodeGraph 实现**：调用图怎么生成？Go 可用 tree-sitter 或 go/ast；调用关系 + 影响范围分析；token-hub 用 CodeGraph 在"提架构方案前验证影响范围"
5. **多 AI 工具协同**：面试官会问"为什么不只用一个 Claude？"——答：能力互补（Claude 深度分析、Trae 编码、CodeGraph 静态验证、Understand-Anything 索引），且单一工具有知识盲区
6. **Git Hooks vs CI**：Git Hooks 能绕过（`--no-verify`），CI 是最终防线；retrosys 两者都有
7. **文档与代码不同步**（token-docs G13 9%）：怎么发现？token-hub 的做法是"实现后核对源头"+ CodeGraph 审计 + 完成度回核报告；但执行靠人就会漏
8. **多角色评审的价值与代价**：四角色/8 角色评审能发现单视角漏掉的问题，但成本高；适合关键模块，不适合所有 PR

**结合项目怎么答**
- 四防线验收：讲防御纵深（计划→CodeGraph→CR→部署审批），任意一环松懈就漏
- G13 完成度 9%：讲"建了规则但执行不到位"的教训，体现对工程化局限的反思

### 领域 6：需求工程

**要点**
- REQ/PROD/术语表三层文档
- 编码准入条件
- 需求评审状态机
- 需求挖掘 CLI

**盲点**
1. **需求准入的代价**（retrosys 8 需求仅 1 准入）：面试官会质疑"太慢了，是不是过度工程？"——答：需求工程化的成本前置，防止"AI 帮你把错误的需求实现得很漂亮"；但要控制节奏，不能一个选型卡住整条链路
2. **需求工程 vs 敏捷**：重需求工程看似反敏捷，但敏捷不等于"不写需求"；retrosys 是"重前期 + 准入后快速编码"的混合模式
3. **术语表维护**：60+ 条，谁来维护？更新机制？答：需求评审时新增，PR 模板检查术语一致性；但实际容易滞后
4. **需求挖掘 CLI**（retrosys elicit.py）：5 阶段挖掘怎么避免"无尽的需求"？答：validate 阶段做优先级排序 + MVP 范围界定
5. **PROD 和 REQ 的边界**：REQ 是技术语言（开发者看），PROD 是设计语言（产品看）；何时该写 PROD？REQ 评审通过后、编码前
6. **需求关联图**：REQ-001 下游触及 002/005/006/007/008——改一个需求的影响范围能可视化，但维护成本高

**结合 retrosys 怎么答**
- 进度 100% 需求但 0% 数据库：诚实讲"过度规划 vs 编码滞后是 vibe coding 典型陷阱"，体现自我反思
- 准入机制：讲"防止 AI 漫游"，但承认"8 个只 1 个准入"的节奏问题，改进方向是并行推进多个 REQ 评审

### 领域 7：安全

**要点**
- JWT：HS256、过期、刷新
- bcrypt：加盐、cost factor
- API Token：sk-xxx 格式、吊销
- 脱敏：maskKey
- crypto/rand vs math/rand

**盲点**
1. **JWT 吊销**（无状态难题）：JWT 无状态无法主动吊销；方案：短过期（access token 15min）+ 刷新 token + 黑名单（Redis）；token-hub 的做法要问清
2. **JWT 存哪里**：localStorage（XSS 可窃取）vs httpOnly cookie（CSRF 风险）；权衡：httpOnly + SameSite + CSRF token 更安全
3. **bcrypt cost**：cost=10 约 100ms，cost=12 约 400ms；太高影响登录性能，太低易被暴力破解；一般 10-12
4. **API Token 泄露检测**：怎么知道泄露了？异常调用模式（IP/频率/地理位置）；怎么吊销？token 加版本号 + Redis 黑名单
5. **crypto/rand vs math/rand**（token-hub 经典踩坑）：`generateTempPassword()` 用循环索引 `string(runes[i%len])` 而非 `crypto/rand` → 所有临时密码固定 `abcdefghij`；这是安全编码的反面教材，面试讲从踩坑到建安全规范的成长
6. **时序攻击**：字符串比较 `==` 或 `strings.EqualFold` 有时序差，攻击者能逐字节猜；密码/token 比较要用 `crypto/subtle.ConstantTimeCompare`
7. **CSRF 防护**：双通道认证（retrosys）怎么防 CSRF？SameSite cookie + CSRF token + 自定义头校验
8. **SQL 注入**：GORM 默认参数化安全，但 `Raw()` + 字符串拼接危险；拼接必须用 `?` 占位
9. **脱敏的全面性**：日志（maskKey）、响应（部分字段）、错误信息、监控面板、数据库备份——每个路径都要脱敏，漏一个就泄露
10. **密钥轮换**：JWT secret 怎么轮换？双 secret 并存期 + 逐步切换；API Key 怎么 rotate？老 key 宽限期 + 新 key 生效

**结合 token-hub 怎么答**
- generateTempPassword 踩坑：讲"用循环索引生成密码 → 固定 abcdefghij"，从踩坑到引入 crypto/rand + 安全编码规范，体现安全意识成长
- JWT + API Token 双层：讲 JWT（用户会话）+ API Token（程序调用 sk-xxx）的分工

### 领域 8：运维部署

**要点**
- Docker：多 Dockerfile、多阶段构建
- Nginx：反向代理、SSE 代理
- SSH 免密
- Redis：缓存、持久化

**盲点**
1. **Docker 多阶段构建**：builder 阶段编译 → 运行阶段只拷二进制，镜像从几百 MB 降到几十 MB；token-hub 有 4 个 Dockerfile（backend/frontend/portal/总）
2. **Nginx SSE 代理**：必须 `proxy_buffering off` + `proxy_cache off` + 长超时 `proxy_read_timeout 3600s`；否则 Nginx 缓冲会导致流式失效
3. **Redis 持久化**：RDB（快照、恢复快、可能丢数据）vs AOF（日志、丢数据少、文件大）；生产一般 RDB + AOF 混合
4. **Redis 降级策略**（token-hub G04a P0）：Redis 挂了计费怎么办？降级到本地缓存 + 拒绝写操作？还是继续用 DB？必须有预案
5. **生产回滚**：蓝绿部署（切流量瞬间回滚）vs 金丝雀（逐步放量发现 问题）；数据库回滚最难（迁移不可逆）
6. **日志收集**：token-hub 用 Zap + 文件 + 内存（trace_store 环形缓冲区）；生产一般 Zap → file → ELK/Loki
7. **健康检查**：`/health` 端点 + Docker HEALTHCHECK + Nginx 被动检查；K8s 有 liveness/readiness 探针
8. **配置管理**：config.yaml + 环境变量覆盖；retrosys 用 pydantic-settings 24 字段 + feature flag（enable_akshare_collection 等）

**结合 token-hub 怎么答**
- G04a Redis 降级缺失：讲 P0 教训，"Redis 挂了计费模块没有降级路径，要么继续写但数据不一致，要么拒绝服务"，改进方向是降级到 DB + 异步补偿
