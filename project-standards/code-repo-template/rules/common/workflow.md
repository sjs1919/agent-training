# 任务工作流

## 主流程

```
需求澄清 → 需求变更影响分析 → 任务拆分 → 9角色评估 → 计划验证 → 用户确认 → 执行自验 → 代码验收 → CR走读 → 部署
  待确认        待确认          待确认      🔴门禁      🔴门禁1     人工       每任务核对   🔴门禁2    🔴门禁3   🔴门禁4
```

**每个环节审查前必须回核上一环节输入**：任务拆分前重读需求澄清结论，9 角色评估前重读任务拆分，计划验证前重读 spec 验收标准，代码验收前重读 plan 任务，CR 走读前重读 spec+plan。**禁止未回核即下结论**，回溯时发现上游遗漏必须补回上游，不得跳过。

- **阶段0（强制）**：不清晰的需求禁止进入阶段1。输出 `analysis-功能名-日期.md`
  - **有 UI 原型时**：可使用 Superpowers-zh Brainstorming 视觉伴侣协助界面设计（详见 `common/skill-integration.md`）
- **阶段1 — 需求变更影响分析（强制·涉及现有代码修改时）**：确认需求变更波及范围，输出影响分析报告 `impact-功能名-日期.md`
  - **涉及现有功能修改时**：使用 CodeGraph 语义知识图谱 + Understand-Anything 交互式知识图谱进行影响分析（详见 `common/skill-integration.md`）
  - 输出变更点清单（逐文件/逐符号），用于阶段2 任务拆分
- **阶段2 — 任务拆分**：按 Phase 编号（`T1.1`、`T1.2`），写入文档仓 `05-tasklist/todo-功能名-日期.md`（纯状态索引，验收细节以 plan 为唯一源）

### 拆分粒度

- 按层级分组为 Phase，自底向上
- 每个任务 ≤ 3 个变更点，每任务聚焦一个文件/一个模块
- 独立文件任务可并行（≤ {{MAX_PARALLEL_AGENTS}}），其余串行
- **Phase 门禁**：每个 Phase 结束必须 `{{BUILD_CMD}}` 构建通过 + 相关测试通过 + 回核 spec 该 Phase 涉及的全部验收标准

### 9 角色评估（计划进入执行前强制）

| 角色 | 关注点 | 不合格即驳回 |
|------|--------|-------------|
| 开发者 | 代码正确性、构建通过、变更点逐个覆盖 | 有遗漏变更点 |
| 测试者 | 单元测试覆盖、E2E 链路完整、边界条件 | 无测试任务 |
| 架构审查 | 分层清晰、依赖方向正确、无循环引用 | 跨层反向依赖 |
| 安全审查 | SQL 注入、金额精度、敏感信息泄漏、XSS/CSRF/SSRF、认证授权、越权（垂直/水平）、密钥与凭证管理、加密算法选型、依赖漏洞（CVE）、上传文件校验、限流防爆破 | 有安全隐患 / 凭证硬编码 / 未鉴权接口 / 越权可读写他人数据 |
| DBA 运维 | migration 幂等、回滚路径、列类型匹配 | 无回滚方案 |
| 前端架构师 | 组件分层（视图/容器/逻辑/UI 原子）、状态管理边界、路由/权限守卫、API 层抽象、构建产物体积、跨端兼容、设计系统一致性 | 业务逻辑写进 UI 组件 / 全局状态污染 / 路由权限旁路 |
| 前端开发工程师 | 类型完备、响应式正确性、表单校验、加载/空/错态、可访问性 a11y、i18n、性能（懒加载/虚拟列表/防抖节流）、浏览器兼容、样式作用域 | 类型兜底逃逸 / 无 loading 与错误处理 / 直接 DOM 操作绕过框架 / 金额精度处理不当 |
| 代码走读 | 命名一致性、注释/文档同步、死代码/重复代码、TODO/FIXME 收口、复杂度（圈复杂度/嵌套层级）、错误处理传播路径、日志级别合理性、是否遵守项目编码规则（CLAUDE.md/rules 红线） | 存在 TODO 未登记 / 错误被吞掉 / 违反项目红线 |
| 法律合规审查 | ICP/数据出境/个人信息保护法（PIPL）、用户协议与隐私政策、第三方 SDK/开源许可证（GPL/AGPL 传染）、计费票据合规、敏感词与内容审核、未成年人保护、API 调用上游 ToS 合规 | 引入 AGPL/SSPL 组件 / 缺隐私政策 / 跨境数据传输无合规依据 / 计费无审计凭证 |

## 四门禁 + 每任务核对

- **🔴 门禁1 — 计划验证（强制）**：计划写完后打开 spec 逐条核对——spec 每条验收标准在计划中有对应任务？计划末尾有端到端验收任务？9 角色评估全部 PASS？遗漏任何一条禁止进入用户确认。
- **阶段3 — 执行**：一次聚焦一个任务，做完自验后再执行下一个；发现阻塞立即记录；**禁止只执行一半就登记完成**。
- **🔴 任务完成核对（强制·每个任务）**：
  1. 打开 spec，逐条读出本任务对应的验收标准，逐条确认已实现
  2. 运行测试并**阅读输出**，确认 PASS 数量符合预期，不能只看「没有红色」
  3. `{{BUILD_CMD}}` 构建通过
  4. **迁移/重写类任务**：打开原实现文件，逐字段/逐按钮/逐调用对比新实现，禁止凭感觉写
  5. 向用户报告核对结果：`spec §x 第 A/B/C 条 → ✅/❌ | 测试 X pass / Y fail | 构建 ✅/❌ | 未完成项：无/[列表]`
  6. 全部 ✅ 才能进入下一个任务，禁止带病推进
- **🔴 门禁2 — 代码验收（强制）**：使用代码审查工具（如有 codegraph、Understand-Anything 等）阅读改动相关代码，逐项对照 plan 任务确认每个步骤已实现 → 打开 spec 逐条核对验收标准 → 运行全量测试（`{{TEST_CMD}}` + `{{BUILD_CMD}}`）→ 运行 plan 末尾端到端验收（默认执行文档仓 `08-test/` 冒烟脚本）→ 报告「spec 共 N 条，已实现 M 条，未实现 0 条」。**禁止跳过实际代码审查直接声称已完成**，任何方案讨论、设计建议、验收结论必须以实际代码/spec/数据为依据，不得凭感觉空谈。
- **🔴 门禁3 — CR 批判式走读（强制·最后一道防线）**：以批判者视角逐行（有没有 bug？边界条件？错误处理？）、逐函数（职责单一？参数合理？副作用？）、逐文件（命名一致？无用引入清理？）审阅 diff；可使用 Understand-Anything `/understand-diff` 辅助分析变更波及面；发现问题 → 修复 → 重新走读，直到无问题；**走读不通过禁止提交/合并/部署**。
- **🔴 门禁4 — 部署审批（强制·无文档禁止部署）**：
  1. **部署文档**：每次部署前必须先产出部署文档（写入 `10-deployment/`，结构见 `common/deployment.md`），未写或结构不完整禁止进入下一步
  2. **用户审核**：部署文档提交用户审核，审核通过并签字后方可执行部署
  3. **备份→部署→验证→回滚**：全部按部署文档中预先写好的命令序列执行，禁止临时拼凑
  4. **验证清单全 PASS**：部署后跑完整链路验证，任一项 FAIL 则立即回滚
  5. AI 不得在任何阶段自主触发部署操作

## 执行模式两挡

### 确认挡（默认）

现行流程，每阶段人工确认。

### 自动挡（触发词启动，如「自动挡执行」）

- **前置条件**（三项全满足才允许进入）：① spec 已用户确认（门禁②）② plan 已用户确认（门禁③）③ plan 中全部编码类任务带「自动验收命令」
- **运行规则**：逐任务执行 → 跑自动验收（测试/构建/grep/脚本）→ PASS 记录证据进下一任务；FAIL 修复重试，同一错误 ≥5 次触发死循环停问（复用 conversation.md 规则）
- **四门禁照跑不问人**：AI 自检完成并输出证据日志——门禁1 计划核对记录、门禁2 验收报告（spec 逐条 ✅/❌ + 测试输出）、门禁3 CR 走读记录
- **硬中断红线**（命中必须停下问人，自动挡不豁免）：① 服务器写操作 ② 任何部署动作 ③ 破坏性 git 操作（force push / reset --hard / 删分支）④ 发现 spec 内部矛盾或验收标准不可自动判定
- **5 个人工门禁**（任何模式不可跳过）：① 给需求 ② 需求确认（spec 确认）③ 计划确认 ④ 最终测试（人工按 spec「最终人工测试清单」验证）⑤ 门禁4 部署审批（部署文档审核 + 签字）
- **终点交付**：全任务 PASS + 四门禁证据日志 + deploy-checklist 第一段填写完毕 → 移交人工终测（门禁④）→ 终测通过 → 门禁4 部署审批（先写文档，审核通过后执行）

## 技能集成

工作流中三个环节可联动外部 Skill —— 详见 `common/skill-integration.md`：

- **阶段0 · 需求澄清**（有 UI 原型时） → Superpowers-zh Brainstorming 视觉伴侣
- **阶段1 · 需求变更影响分析**（涉及现有功能修改时） → CodeGraph + Understand-Anything
- **门禁2 · 代码验收** → CodeGraph 辅助代码审查
- **门禁3 · CR 走读** → Understand-Anything `/understand-diff`
