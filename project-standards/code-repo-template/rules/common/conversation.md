# 对话与提问规范

## 一次一问

- ✅ 聚焦一个话题 → 说明「为什么问」→ 给选项 → 等确认 → 下一个
- ❌ 一条消息混合「确认+新问题+建议」、一次发 3+ 问题
- 需求澄清逐层展开：配置/需求讨论一次只展开一个问题，用户充分澄清后再进入下一个

## 死循环停问（重试 ≥5 次）

相同指令/相同错误重试超过 5 次仍失败 → 立刻停下来向用户报告，禁止死犟。

## 遇问题一次失败即问

环境类问题（代理、端口、权限、依赖安装等）一次失败后即问用户，不反复尝试。

## 带方案来问

停下来询问时必须同时提供「问题分析 + 至少一个解决方案」，禁止空手来问。

- ❌ 「这个问题我试了 5 次不行，请你看看」
- ✅ 「这个问题试了 5 次，原因是 XXX。方案 A… 方案 B… 你选哪个？」

## 带行业对标来探讨

讨论方案时，优先寻找行业通用做法、竞品参考、开源项目先例作为参照，避免闭门造车。

- ✅ 「权限模型业界三种主流做法：RBAC（大多数 SaaS）、ABAC（AWS IAM）、ReBAC（Google Zanzibar）。我们场景是 X，建议 RBAC，理由是 Y」
- ❌ 不提行业参照，仅凭个人经验下结论

## 务实可挑战

每个角色提出的审查意见必须满足两条底线：

1. **务实** — 每条意见必须对应具体代码/数据/可重现步骤，禁止空泛批评（「这个设计不好」缺少具体论据）
2. **经得起挑战** — 每条意见随时可被其他角色追问「为什么」，回答必须有推理链条。裁决策略：**互相不能说服时，以更安全/更简单的一方为准**

## 禁止替用户做选择

提出的方案/建议 ≠ 用户的答案。用户未明确选定的问题，不得在后续文档、设计、代码中默认采用；必须等用户明确选择，确认后才能在产出物中标记为「已确认」。

- ❌ 「我已经把方案 A 写到文档里了」
- ✅ 「方案 A/B/C 你选哪个？」→ 等用户选 → 确认后写入

## Bug 修复逐个验收

修复一个 bug → 立即验收通过 → 再修下一个。禁止批量修复多个 bug 后一次性验收。
