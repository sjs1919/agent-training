# 前后端 API 契约规范

## 核心定位

文档仓 `03-design/api-spec.md` 是前后端的**接口契约（Source of Truth）**：

- 后端行为必须与 api-spec.md 对齐——文档说返回什么就返回什么
- 前端对接必须以 api-spec.md 为准——文档没有的接口**禁止凭空捏造**
- 契约优先于实现：文档一旦锁定，**后端不符合文档 = 后端 bug**
- curl 验证是发现不一致的检测手段，不是替代文档；发现不一致 → 修后端或修文档 → 必须对齐

## 版本号

api-spec.md 头部必须含版本号 `vYYYY.MM.DD`；每次修改文档同步更新版本号；前端对接前先核对版本号。

## 接口生命周期

**后端主导写初稿，前端逐字段 review 确认后锁定，双方并行开发。**

微调 vs breaking 口诀：**前端旧代码不改就能跑的 = 微调；前端不改就报错的 = breaking。**

| 变更类型 | 分类 | 流程 |
|---------|------|------|
| 新增可选字段 / 可选参数 / 新增接口 | 微调 | 改代码 + 文档即可 |
| 字段类型修改 / 重命名 / 删除、新增必填参数、删除接口、修改成功判断码 | **breaking** | 先改文档 → 前端确认 → 再改代码 |

不确定时默认按 breaking 处理。

## 🚫 后端编码完毕 → 必须对照更新 api-spec.md

**触发条件（任一即触发）**：新增/修改/删除路由；修改响应字段名/类型/增删；修改请求参数；新增错误码。

**强制流程**：编码完成 → 打开 `03-design/api-spec.md` 逐接口对照 → 文档与代码不一致则更新文档 → 更新版本号 `vYYYY.MM.DD` → 提交时文档与代码同包。**禁止代码合入后文档还停留在旧版本。**

## 🚫 前端对接后端接口 → 必须先查 api-spec.md

**强制流程**：需要调用后端接口 → 先打开 `03-design/api-spec.md` 搜索该接口：

1. **文档已有** → 严格按文档的路径/方法/字段名/类型写，不做任何假设
2. **文档没有** → 🚫 禁止凭空捏造！必须先让后端补文档，或 curl 验证实际响应并记录快照

**curl 验证**：不一致时以实际响应为准先修复前端，同时通知后端更新文档到一致。禁止前端绕过文档直接「猜」接口。

## 核对清单（前端对接完成时逐条执行）

| # | 核对项 | 示例错误 |
|---|--------|---------|
| 1 | 请求路径 | `/token/list` vs `/user/tokens` |
| 2 | 请求方法 | GET vs POST |
| 3 | 请求参数（全部字段名和类型） | `old_password` vs `oldPassword` |
| 4 | 响应字段名和类型（含嵌套） | `prefix` vs `token_prefix` |
| 5 | 分页包裹字段名 | `data.list` vs `data.items` |
| 6 | 成功判断码 | `code: 200` vs `code: 0` |
| 7 | 前端类型定义与文档一致 | 字段名/类型/可选性 |
| 8 | 字段约束（枚举值、必填/可选、起止值） | 枚举未定义 / page 从 0 还是 1 |

字段表格规范：每个接口必须列出字段表（字段/类型/必填/说明）；枚举值列全；金额注明单位；分页注明起始值。

## 错误码一致性

前后端错误码与文档仓 `03-design/api-spec.md` 对齐；前端统一判断 `code === 200` 为成功；**新增错误码 → 同步登记 api-spec.md §4**。

## 禁止事项

- 🚫 未对照文档或 curl 验证就凭空假设路径/字段名
- 🚫 后端改接口只改代码不更新文档
- 🚫 前端发现不一致只修前端不修文档
- 🚫 跳过前端 review 直接锁定接口文档
