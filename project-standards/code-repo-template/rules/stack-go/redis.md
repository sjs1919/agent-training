# Redis 规则

## 部署前提

- 默认按 `standalone` 部署设计；此前提下允许用逻辑 DB 做业务隔离
- 若调整为 Cluster 或托管方案（不支持多逻辑 DB），必须先更新本规则与相关 spec

## 设计原则

- Redis 只承载缓存、限流计数、短时会话态和多实例共享运行态，**不作为业务数据的最终事实来源**
- **按业务域分逻辑 DB**：不同一致性要求、热点特征和 TTL 策略的数据禁止混放同一 DB。示例：
  - 缓存域（如目录/权限类热点读）与限流计数域（最强热点写）必须分库，避免互相争抢资源
  - 短 TTL 会话态单独一库，允许丢失或自然过期，不污染业务计数库
- 新增 Redis 场景必须先定义：所属业务域、key 前缀、TTL、容量预估、失效方式、故障降级策略，再允许落地
- 读路径默认「优先读 Redis，失败或未命中回源 DB/本地状态」；写路径默认「先写主存，再删缓存或刷新缓存」
- **高并发计数必须使用 Redis 原子能力，禁止 `GET` 后本地加减再 `SET` 的非原子写法**

## key 规范

- 统一业务域命名：`{domain}:{biz_key}`；推荐前缀示例：`cache`、`perm`、`rl`、`quota`、`session`、`auth`、`metrics`
- key 名必须体现业务主键，禁止无语义缩写；优先使用稳定 ID 或规范化名称
- 批量操作必须基于前缀、集合索引或显式 key 列表，禁止线上无边界全量扫描作为常规业务路径

## TTL 与一致性

- 缓存类必须设置 TTL，并提供写后失效或刷新机制
- 限流与配额计数的 TTL 必须与业务窗口严格一致
- 涉及权限关闭、禁用、额度归零等风险控制场景，必须优先保证快速失效

## 降级要求

- Redis 不可用时，缓存类必须允许回源 DB 或本地状态，不能直接阻断核心链路
- 限流计数依赖 Redis 共享状态时必须定义降级策略：回退本地限流、只读保护或临时收紧限额，**禁止静默放开**
- Redis 超时、拒绝连接、脚本失败、热点 key 异常时必须输出可观测日志与监控指标

## 禁止项

- 禁止把主库表完整镜像到 Redis 作为长期副本
- 禁止把明细/审计日志等长尾数据堆入 Redis 常驻保存
- 禁止多个业务域共用同一组 key 前缀
- 禁止硬编码临时 key 规则而不补充到 spec 或本规则
