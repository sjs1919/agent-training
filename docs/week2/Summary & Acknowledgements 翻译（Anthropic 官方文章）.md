# Summary \& Acknowledgements 翻译（Anthropic 官方文章）

## 摘要（Summary）

在大语言模型（LLM）领域取得成功，关键不在于搭建最复杂的系统，而在于打造适配自身需求的系统。从简单的提示词入手，通过全面的评测对其进行优化，只有当简单方案无法满足需求时，再逐步增加多步骤智能体系统。

在实现智能体时，我们遵循三大核心原则：

1\. 保持智能体设计的简洁性；

2\. 优先保证透明性，清晰展示智能体的规划步骤；

3\. 通过完善的工具文档和充分的测试，精心打造智能体与计算机的交互界面（ACI）。

各类框架能帮助你快速上手，但在向生产环境迁移时，不妨大胆简化抽象层级，使用基础组件进行构建。遵循这些原则，你打造出的智能体不仅功能强大，还具备可靠性、可维护性，能够获得用户的信任。

## 附录1：智能体的实际应用（Appendix 1: Agents in practice）

我们与客户的合作过程中，发现了两个极具前景的 AI 智能体应用场景，充分印证了前文所述设计范式的实用价值。这两个场景均体现出，智能体在兼具对话与操作需求、具备清晰成功判定标准、支持反馈闭环且融入有效人工监督的任务中，能发挥最大价值。

### A\. 客户支持（Customer support）

客户支持场景将大家熟悉的聊天机器人界面与工具集成能力相结合，实现功能增强。这类场景天然适配开放式智能体，原因如下：

1\. 客户支持交互天然遵循对话流程，同时需要获取外部信息并执行相关操作；

2\. 可集成工具以调取客户数据、订单历史及知识库文章；

3\. 退款办理、工单更新等操作可通过程序自动处理；

4\. 可通过用户定义的解决标准，清晰衡量任务完成效果。

已有多家企业通过基于使用量的定价模式验证了该方案的可行性——仅对成功解决问题的场景收费，这也体现出他们对自身智能体效能的信心。

### B\. 编码智能体（Coding agents）

软件开发领域展现出大语言模型功能的巨大潜力，其能力已从代码补全逐步发展为自主解决问题。智能体在该领域表现尤为突出，原因如下：

1\. 代码解决方案可通过自动化测试进行验证；

2\. 智能体可将测试结果作为反馈，对解决方案进行迭代优化；

3\. 问题场景边界清晰、结构规整；

4\. 输出质量可通过客观标准进行衡量。

在我们的实际实现中，智能体如今仅通过拉取请求（pull request）描述，就能解决 SWE\-bench Verified 基准测试中的真实 GitHub 问题。不过，尽管自动化测试有助于验证功能，但人工审核仍是确保解决方案符合更广泛系统需求的关键环节。

## 附录2：工具的提示词工程（Appendix 2: Prompt engineering your tools）

无论你搭建何种智能体系统，工具都可能是其中的重要组成部分。通过在 API 中明确工具的具体结构和定义，可让 Claude（Anthropic 旗下大模型）与外部服务及 API 进行交互。当 Claude 做出响应时，若其计划调用工具，会在 API 响应中包含工具使用模块。工具定义和规范的优化，应与整体提示词的工程优化给予同等重视。在本简短附录中，我们将介绍如何对工具进行提示词工程优化。

实现同一操作通常有多种方式。例如，你可以通过编写差异文件（diff）或重写整个文件来指定文件编辑操作；对于结构化输出，可将代码嵌入 Markdown 格式或 JSON 格式中。在软件工程中，这些格式差异仅为表面形式，可实现无损转换。但对大语言模型而言，部分格式的编写难度远高于其他格式——编写差异文件需要在写入新代码前，明确代码块头部待修改的行数；而在 JSON 中编写代码（相较于 Markdown），则需要对换行符和引号进行额外转义。

关于工具格式的选择，我们给出以下建议：

1\. 给模型预留足够的 Token，使其在陷入格式困境前有足够的“思考”空间；

2\. 工具格式尽量贴近模型在互联网文本中常见的自然格式；

3\. 确保格式无额外“操作负担”，例如无需精准统计数千行代码的行数，或对编写的任意代码进行字符串转义。

一个经验法则是：参考人机交互界面（HCI）的设计投入，在打造优质智能体与计算机交互界面（ACI）时，投入同等的精力。以下是具体的实施思路：

1\. 换位思考，站在模型的角度思考：仅通过工具描述和参数，是否能清晰了解工具的使用方法，还是需要仔细琢磨？如果需要琢磨，那么模型大概率也会面临同样的问题。一份优质的工具定义应包含使用示例、边界情况、输入格式要求，以及与其他工具的清晰边界。

2\. 如何修改参数名称或描述，使其含义更清晰？可将其类比为为团队中的初级开发人员编写优质的文档字符串，尤其是在使用多个相似工具时，这一点尤为重要。

3\. 测试模型对工具的使用情况：在工作台中输入大量示例，观察模型易犯的错误，并进行迭代优化。

4\. 对工具进行防错设计（Poka\-yoke）：调整参数设置，降低模型出错的概率。

在为 SWE\-bench 搭建智能体的过程中，我们实际上在工具优化上投入的时间，远超对整体提示词的优化。例如，我们发现当智能体离开根目录后，在使用相对文件路径的工具时容易出错。为解决这一问题，我们修改了工具设置，强制要求使用绝对文件路径——结果发现，模型此后能完美使用该工具。

deepagents 是一个独立的库，构建于 LangChain 智能体核心构建模块之上。它采用 LangGraph 运行时，可实现持久化执行、流式传输、人机协同循环等功能。

## Interpreters（解释器）

在智能体系统中，解释器（Interpreters）是连接智能体推理逻辑与外部工具、执行环境的核心组件，负责解析智能体输出的行动指令，将其转换为可被计算机执行的操作，并将执行结果反馈给智能体，是智能体实现“推理\-行动”闭环的关键环节。

结合前文提及的智能体架构与工具提示词工程，解释器的核心作用主要体现在以下方面：

1\. 指令解析：精准识别智能体输出的工具调用指令（如工具名称、参数、执行方式），兼容前文提到的各类工具格式（如 JSON、Markdown），处理指令中的格式转义、参数校验等问题，避免因格式错误导致工具调用失败。

2\. 跨环境适配：适配不同的执行环境（如本地开发环境、云端生产环境、代码运行沙箱），将解析后的指令转换为对应环境可执行的代码或命令，例如将智能体生成的代码片段，通过解释器在指定沙箱中运行，适配编码智能体的自动化测试与迭代需求。

3\. 结果封装与反馈：收集工具或执行环境的输出结果（包括成功响应、错误信息），按照前文附录中推荐的结构化规范（如 JSON 格式）进行封装，补充执行状态、结果可信度等元数据，反馈给智能体供其后续推理使用，助力智能体实现迭代优化。

4\. 异常处理与兜底：结合智能体的边界防护原则，解释器可拦截非法指令、异常执行结果，进行错误提示或自动降级处理（如调用备用工具、返回标准化错误信息），降低智能体因指令解析或执行失败导致的流程中断风险，提升系统稳定性。

在 LangChain、LangGraph 及 deepagents 生态中，解释器与智能体、工具集协同工作：LangChain 提供基础的指令解析能力，LangGraph 运行时通过解释器实现智能体行动的持久化执行与流式传输，deepagents 则可基于这些基础组件，定制适配特定场景（如客户支持、编码）的解释器，优化指令解析效率与执行兼容性。

## 智能体后端组件详解（StateBackend 及各类衍生后端）

在 LangChain、LangGraph 及 deepagents 智能体生态中，各类后端（Backend）是支撑智能体持久化执行、状态管理、环境交互的核心底层组件，主要解决智能体运行过程中的状态存储、操作执行、上下文管理等问题，保障智能体“推理\-行动”闭环的稳定性与连续性。以下对各类核心后端进行全面解析，明确其作用、运行流程及适用场景。

### 一、StateBackend（状态后端）

StateBackend 是所有后端组件的基础核心，也是智能体状态管理的核心抽象层，定义了智能体运行状态的存储、读取、更新规范，是实现智能体持久化执行、断点续跑、流式传输的基础。

1\. 核心作用：负责管理智能体运行过程中的全量状态数据，包括智能体的推理步骤、工具调用记录、执行结果、上下文信息、循环次数等，避免因程序重启、网络中断导致状态丢失，支撑 LangGraph 运行时的“持久化执行”核心功能。

2\. 运行流程：① 智能体启动后，StateBackend 初始化状态存储容器（内存、文件、数据库等）；② 智能体每完成一轮推理或工具调用，解释器将当前状态数据（如行动指令、执行结果）同步至 StateBackend；③ 当智能体需要继续执行（如断点续跑、多轮循环）时，从 StateBackend 中读取历史状态数据，恢复至中断前的运行状态；④ 智能体执行终止后，可选择将最终状态数据持久化存储，用于后续日志排查、效果复盘。

3\. 核心用途：适配智能体长流程运行、断点续跑场景，解决“智能体执行中断后需重新开始”的问题，保障执行连续性；同时为智能体的流式传输提供状态支撑，实现推理步骤、执行结果的实时推送。

4\. 适配场景：所有需要持久化执行、状态回溯的智能体场景，如多步骤编码智能体、长流程客户支持智能体、需要人机协同循环的智能体系统。

### 二、FilesystemBackend（文件系统后端）

FilesystemBackend 是 StateBackend 的具体实现之一，基于本地文件系统实现智能体状态的持久化存储，是最基础、易部署的状态存储后端。

1\. 核心作用：将智能体的状态数据以文件形式（如 JSON、YAML）存储在本地磁盘，实现状态数据的持久化保存，支持状态的读取、更新、删除操作，无需依赖外部数据库，部署成本极低。

2\. 运行流程：① 初始化时，指定本地文件存储路径（如指定文件夹路径），创建状态存储目录；② 智能体运行过程中，将状态数据序列化为 JSON 格式，写入指定路径下的文件（可按智能体 ID、执行批次命名文件）；③ 读取状态时，从指定文件中读取数据并反序列化为可识别的状态对象；④ 更新状态时，覆盖原有文件内容或追加写入新的状态数据；⑤ 执行终止后，状态文件可长期保留，用于后续复盘、调试。

3\. 核心用途：本地开发调试、小型智能体项目、对状态存储要求不高的场景，快速实现智能体状态持久化，无需配置复杂的外部存储服务。

4\. 优缺点：优势是部署简单、无需依赖外部服务、成本低；劣势是不支持分布式部署，仅适用于单节点运行的智能体，状态文件易因磁盘故障丢失，不适合生产环境大规模使用。

### 三、LocalShellBackend（本地终端后端）

LocalShellBackend 是面向本地终端环境的操作执行后端，主要负责解析智能体输出的终端操作指令，在本地终端环境中执行，并将执行结果反馈给智能体，是编码智能体、本地操作类智能体的核心交互后端。

1\. 核心作用：连接智能体与本地终端环境，解析智能体生成的终端命令（如代码运行、文件操作、依赖安装等），在本地终端中执行并捕获执行结果（包括输出内容、错误信息、执行状态），反馈给智能体供其后续推理优化。

2\. 运行流程：① 智能体结合任务需求，生成本地终端操作指令（如“python test\.py”“pip install langchain”），经解释器解析后传递给 LocalShellBackend；② 后端校验指令合法性（如拦截高危命令），在本地终端中执行该指令；③ 捕获终端的执行输出、错误日志及执行状态（成功/失败），按结构化格式封装；④ 将封装后的执行结果反馈给解释器，再由解释器传递给智能体；⑤ 智能体根据执行结果，判断是否需要调整指令、重新执行或终止流程。

3\. 核心用途：本地开发场景下的编码智能体、本地文件操作智能体、终端命令执行类智能体，实现智能体对本地终端的自动化操作，适配前文提及的编码智能体自动化测试、代码迭代需求。

4\. 关键特性：支持指令合法性校验、高危命令拦截（如 rm \-rf），保障本地环境安全；可捕获详细的执行错误信息，助力智能体迭代优化指令；仅适用于本地环境，不支持云端远程执行。

### 四、StoreBackend（存储后端）

StoreBackend 是通用的持久化存储后端抽象，可适配多种存储介质（如数据库、缓存、云存储），核心用于存储智能体运行过程中的非状态类数据，与 StateBackend 形成互补，支撑智能体的上下文管理、工具数据存储等需求。

1\. 核心作用：存储智能体运行过程中需要长期保留、可复用的非状态数据，包括知识库文档、工具定义、用户配置、历史对话记录、执行日志等，实现数据的统一管理与复用，避免重复存储、重复获取。

2\. 运行流程：① 初始化时，指定存储介质（如 SQLite、Redis、阿里云 OSS），配置存储连接参数；② 智能体运行过程中，需要存储数据时（如检索到的知识库 chunk、工具定义信息），通过解释器传递给 StoreBackend；③ 后端将数据按指定格式（如 JSON、二进制）存储至对应介质，生成唯一标识用于后续读取；④ 智能体需要获取数据时，通过唯一标识从 StoreBackend 中读取对应数据；⑤ 支持数据的更新、删除、筛选操作，适配不同场景的数据管理需求。

3\. 核心用途：智能体知识库存储、工具定义管理、用户对话历史留存、执行日志归档，适配客户支持智能体的知识库调取、编码智能体的代码片段存储等场景，与 StateBackend 配合，实现智能体“状态\+数据”的全量管理。

4\. 适配场景：各类需要长期存储可复用数据的智能体场景，支持本地存储与云端存储适配，可根据生产环境需求灵活切换存储介质。

### 五、ContextHubBackend（上下文中心后端）

ContextHubBackend 是专注于智能体上下文管理的后端组件，基于 StoreBackend 扩展而来，核心解决智能体上下文信息的存储、检索、整合问题，支撑智能体多轮对话、多步骤推理的上下文关联需求。

1\. 核心作用：专门管理智能体的上下文信息，包括用户输入、智能体推理步骤、工具执行结果、历史对话记录等，支持上下文的快速检索、筛选、整合，确保智能体在多轮循环中能准确获取历史上下文，避免上下文丢失或混淆。

2\. 运行流程：① 智能体每一轮执行完成后，解释器将当前上下文信息（如用户提问、智能体推理内容、工具反馈结果）传递给 ContextHubBackend；② 后端对上下文信息进行结构化处理（如按对话轮次、执行步骤分类），并存储至指定介质；③ 智能体进行下一轮推理时，根据当前任务需求，从 ContextHubBackend 中检索相关历史上下文（如筛选近3轮对话记录）；④ 后端将检索到的上下文信息整合后，反馈给智能体，辅助其完成推理判断；⑤ 支持上下文的过期清理、优先级排序，避免上下文冗余导致的 Token 浪费。

3\. 核心用途：多轮对话智能体、多步骤推理智能体、需要上下文关联的场景，如客户支持智能体的多轮问题解答、编码智能体的多步骤代码编写，解决智能体“遗忘”历史信息的问题，提升推理准确性。

4\. 关键特性：支持上下文的精细化管理（如按优先级、时间筛选）、冗余清理，适配大模型上下文窗口限制；可与解释器协同工作，自动筛选与当前任务相关的上下文信息，减少无效信息干扰。

### 六、CompositeBackend（复合后端）

CompositeBackend 是一种聚合型后端，可将多种不同类型的后端整合为一个统一的后端接口，让智能体在运行过程中，根据不同需求自动调用对应的后端组件，实现“一站式”后端管理，提升智能体系统的灵活性与扩展性。

1\. 核心作用：整合 StateBackend、FilesystemBackend、LocalShellBackend 等多种后端组件，提供统一的调用接口，根据智能体的不同操作需求（如状态存储、终端执行、上下文管理），自动路由至对应的后端组件，无需手动切换后端，简化智能体系统的配置与开发。

2\. 运行流程：① 初始化时，配置需要整合的后端组件（如整合 StateBackend 用于状态管理、LocalShellBackend 用于终端执行、ContextHubBackend 用于上下文管理），并定义路由规则（如“状态存储请求路由至 StateBackend，终端操作请求路由至 LocalShellBackend”）；② 智能体运行过程中，通过解释器向 CompositeBackend 发送请求（如状态存储、终端执行、上下文检索）；③ 复合后端根据预设路由规则，将请求转发至对应的子后端组件；④ 子后端组件执行对应操作，将结果反馈给 CompositeBackend；⑤ 复合后端将结果统一封装后，反馈给解释器，再传递给智能体。

3\. 核心用途：复杂智能体系统、多场景适配的智能体项目，如同时涉及状态持久化、本地终端操作、上下文管理的编码智能体，通过复合后端整合各类子后端，简化系统架构，提升开发与维护效率。

4\. 优势与适配场景：优势是灵活可扩展，可根据业务需求新增、移除子后端组件；统一调用接口，降低开发成本；适配复杂生产环境，可同时满足状态管理、数据存储、环境交互等多种需求，是大规模智能体系统的首选后端架构。

### 七、各类后端协同工作场景（适配智能体系统）

以编码智能体为例，各类后端协同工作流程：① 智能体启动后，StateBackend 初始化状态存储，记录初始运行状态；② 智能体生成代码编写指令，经解释器解析后，由 LocalShellBackend 在本地终端执行代码编写、测试操作；③ 执行结果、代码片段由 StoreBackend 存储，上下文信息由 ContextHubBackend 管理；④ 每一轮执行完成后，StateBackend 更新运行状态，CompositeBackend 统一协调各后端组件，实现数据与状态的同步；⑤ 智能体执行终止后，所有状态数据、执行日志、代码片段通过各后端持久化存储，用于后续复盘与优化。

### 八、Sandbox Backend（沙箱后端）

Sandbox Backend（沙箱后端）是一类专用后端组件，核心特点是为智能体代码运行提供隔离环境，自带独立文件系统和终端命令执行工具，是 deepagents 生态中适配编码智能体、高危操作类智能体的重要后端，可有效规避智能体操作对本地机器的影响。

1\. 核心作用：为智能体提供隔离的运行环境，允许智能体在沙箱内执行文件写入、依赖安装、终端命令运行等操作，且所有操作仅在沙箱内部生效，不会修改本地机器的文件、配置及环境，彻底解决智能体执行高危命令、误操作导致本地环境损坏的问题，同时适配 deepagents 智能体的代码编写、测试、依赖安装等核心需求。

2\. 运行流程：① 初始化 deepagents 智能体时，将沙箱后端配置为智能体的默认后端；② 智能体结合任务需求，生成文件写入、依赖安装、代码运行等指令（如“pip install requests”“touch test\.py”），经解释器解析后传递给沙箱后端；③ 沙箱后端在隔离环境内初始化独立文件系统，执行接收到的指令，同时拦截可能破坏沙箱环境的高危操作；④ 捕获沙箱内的执行结果（包括文件创建状态、命令执行输出、依赖安装结果），按结构化格式封装后反馈给解释器，再传递给智能体；⑤ 智能体根据执行结果迭代优化指令，所有操作均在沙箱内闭环，执行终止后可选择保留沙箱环境用于复盘，或直接销毁沙箱释放资源。

3\. 核心用途：适用于需要让智能体执行文件操作、依赖安装、代码运行，且需保护本地机器环境的场景，如 deepagents 编码智能体的代码编写与测试、第三方工具调用测试、高危命令验证等，是生产环境中智能体执行操作类任务的首选后端之一。

4\. 配置与使用要点：在创建 deepagents 智能体时，只需将沙箱后端实例传入智能体的 backend 参数，即可完成配置，无需额外修改智能体的推理逻辑和工具调用规则。沙箱后端会自动接管智能体的所有环境交互操作，实现隔离运行，同时兼容前文提及的工具提示词工程规范，支持智能体生成的各类指令格式。

5\. 与 LocalShellBackend 的核心区别：两者均支持终端命令执行，但 LocalShellBackend 直接在本地终端运行指令，会对本地环境产生影响；沙箱后端则在隔离环境中运行，所有操作与本地机器完全隔离，安全性更高，更适合生产环境和高危操作场景，也是 deepagents 智能体推荐使用的环境交互后端。

## Skills（智能体技能）

在 deepagents 生态中，Skills（技能）是为智能体赋予新能力、专业知识的核心组件，与工具（Tools）、各类后端协同工作，区别于工具的底层功能支撑，技能更侧重任务层面的详细指引与资源整合，可有效提升智能体的任务执行能力与专业性，同时优化上下文与 Token 占用效率。

1\. 核心作用：为智能体提供完成特定任务所需的详细操作指引、参考信息、模板等资源，弥补智能体在专业领域知识、任务执行流程上的不足；通过“渐进式披露”机制，仅在智能体判定当前任务需要对应技能时才加载相关资源，避免启动时加载所有资源导致的 Token 浪费和上下文冗余，提升智能体推理效率。

2\. 与工具（Tools）的核心区别：工具主要覆盖底层基础功能，如原生文件系统操作、任务规划、终端命令执行等，是智能体与外部环境交互的基础载体；而技能则包含完成具体任务的完整逻辑、细节指引和配套资源（如代码模板、报告模板、操作步骤清单），是对工具功能的高阶封装与补充，帮助智能体更规范、高效地完成复杂任务。例如，工具可实现“文件写入”操作，而“编写标准化测试代码”技能则包含测试代码模板、编写规范、常见问题解决方案，指导智能体利用文件写入工具完成具体任务。

3\. 运行流程：① 开发人员提前编写技能文件，包含任务执行步骤、参考信息、模板等内容，关联对应工具；② 创建 deepagents 智能体时，将技能文件作为参数传入 create\_deep\_agent 函数，完成技能配置；③ 智能体启动后，仅加载核心运行组件，不加载任何技能资源，减少初始上下文与 Token 占用；④ 智能体接收用户任务并进行推理，判断当前任务需要的专业能力，筛选出匹配的技能；⑤ 智能体加载对应技能文件，获取任务执行指引、模板等资源，结合工具和各类后端（如沙箱后端、StateBackend）执行任务；⑥ 任务执行过程中，智能体根据技能指引调用对应工具，通过后端完成操作执行、状态存储、结果反馈，形成“技能指引\-工具调用\-后端执行”的闭环；⑦ 任务完成后，智能体释放已加载的技能资源，等待下一轮任务推理。

4\. 核心用途：适用于提升智能体在特定领域的执行能力，如编写标准化代码、生成规范报告、完成特定业务流程（如客户支持工单处理、RAG 检索调优）等场景；同时通过渐进式披露机制，优化智能体的运行效率，降低 Token 消耗，适配长流程、复杂任务的执行需求。例如，为编码智能体配置“单元测试编写技能”，智能体可在需要编写测试代码时，加载该技能的模板与指引，结合沙箱后端完成测试代码的编写与运行。

5\. 配置与使用要点：在创建 deepagents 智能体时，只需将技能文件或技能实例作为参数传入 create\_deep\_agent 函数，即可完成技能集成，无需修改智能体的核心推理逻辑；技能文件可根据业务需求灵活编写、新增或删除，支持与各类后端（StateBackend、StoreBackend、Sandbox Backend 等）协同工作——技能指引智能体调用工具，后端负责执行操作、存储状态与相关资源，实现能力与底层支撑的深度融合。

6\. 与各类后端的协同关系：技能本身不具备执行能力，需依托各类后端实现落地——通过 StateBackend 存储技能加载状态与执行进度，确保技能调用过程可回溯；通过 StoreBackend 存储技能文件、模板等资源，实现技能资源的复用与管理；通过 Sandbox Backend、LocalShellBackend 等执行技能指引下的工具操作，保障任务执行的安全性与稳定性；通过 ContextHubBackend 管理技能调用过程中的上下文信息，确保技能指引与当前任务的关联性。

## Memory（智能体记忆）

在 deepagents 生态中，Memory（智能体记忆）是用于为智能体补充额外上下文信息的核心功能，通过加载指定的 AGENTS\.md 文件，为智能体提供任务相关的参考资料、执行规范、历史经验等内容，与技能、各类后端协同工作，进一步提升智能体的任务执行准确性与专业性。

1\. 核心作用：通过 AGENTS\.md 文件为智能体补充额外上下文，这些文件可包含智能体执行任务所需的参考信息、执行规范、历史任务案例、注意事项等内容，弥补智能体自身知识库的不足；与技能的“渐进式披露”不同，记忆文件可按需加载，为智能体提供更灵活、更具针对性的上下文支撑，同时可结合各类后端实现记忆内容的持久化与管理。

2\. 核心载体：AGENTS\.md 文件是智能体记忆的核心载体，采用 Markdown 格式编写，支持文本、代码块、列表等多种格式，可清晰呈现各类参考信息与执行规范，适配大模型对自然格式文本的处理习惯，降低智能体解析成本，这也贴合前文工具提示词工程中“选用模型熟悉的自然格式”的原则。

3\. 运行流程：① 开发人员提前编写 AGENTS\.md 文件，写入任务相关的参考信息、执行规范、案例等内容，可根据业务需求编写多个记忆文件，分别对应不同任务场景；② 创建 deepagents 智能体时，将 AGENTS\.md 文件的文件路径作为参数传入 memory 参数，完成记忆功能配置；③ 智能体启动后，可根据当前任务需求，加载对应的 AGENTS\.md 文件，获取额外上下文信息；④ 智能体结合记忆文件内容、技能指引、工具调用能力，协同各类后端完成任务执行——通过 StateBackend 存储记忆文件的加载状态，通过 StoreBackend 实现 AGENTS\.md 文件的持久化存储与管理，通过解释器解析记忆文件中的相关规范，指导工具调用与操作执行；⑤ 任务执行过程中，智能体可随时调取记忆文件中的内容，辅助推理判断与指令优化；⑥ 任务完成后，智能体释放已加载的记忆文件资源，同时可通过 StateBackend 记录记忆文件的使用情况，为后续优化提供参考。

4\. 配置与使用要点：在创建 deepagents 智能体时，只需将一个或多个 AGENTS\.md 文件的文件路径传入智能体的 memory 参数，即可完成记忆功能配置，无需修改智能体的核心推理逻辑；AGENTS\.md 文件可灵活编写、更新，支持动态调整记忆内容，适配业务需求的变化；同时，记忆文件的加载可与技能的加载机制协同，仅在智能体判定当前任务需要时加载对应记忆文件，进一步优化 Token 占用与推理效率。

5\. 与各类后端的协同关系：记忆功能的落地需依托各类后端提供支撑——通过 FilesystemBackend 读取本地 AGENTS\.md 文件，实现记忆内容的加载；通过 StoreBackend 存储 AGENTS\.md 文件及记忆内容的解析结果，实现记忆资源的复用与长期管理；通过 StateBackend 存储记忆文件的加载状态、使用记录，确保记忆调用过程可回溯、可监控；通过 ContextHubBackend 整合记忆内容与当前任务上下文，确保智能体能准确关联记忆信息与任务需求，提升推理准确性。

6\. 与 Skills 的核心区别：技能侧重为智能体提供完成特定任务的详细操作指引与配套资源（如模板、步骤清单），是“怎么做”的具体指导；而记忆（AGENTS\.md 文件）侧重为智能体提供额外的参考信息、执行规范、历史案例等，是“参考什么”的信息补充。两者协同工作，技能指导智能体的操作流程，记忆为智能体提供操作过程中的参考依据，结合各类后端与工具，形成“参考\-指引\-执行\-反馈”的完整闭环，进一步提升智能体的任务执行能力。

## Profiles（配置文件）

在 deepagents 生态中，Profiles（配置文件，核心为 Harness Profile）是可复用的模型专属配置包，用于为不同大模型定制差异化运行配置，实现“配置跟随模型”的效果，无需在每次创建智能体时重复配置，大幅提升智能体的配置效率与模型适配性。

1\. 核心作用：解决不同大模型（如 Claude、GPT）的差异化适配问题，将针对特定模型的配置（如系统提示词后缀、工具描述改写、中间件配置）封装为可复用的配置包，当智能体选择匹配的模型时，该配置包会被自动应用。适用于实现“行为跟随模型而非调用端”的需求，例如为 Claude 定制适配其指令风格的系统提示词后缀，为 GPT 改写工具描述，或为特定模型提供商配置专属中间件。

2\. 核心特性与可配置内容：单个配置文件可承载多种模型专属配置，覆盖智能体运行的核心环节，具体包括：① 自定义基础系统提示词（base\_system\_prompt），为特定模型设置专属基础提示词；② 追加系统提示词后缀（system\_prompt\_suffix），在基础提示词后补充适配模型的额外要求；③ 工具描述覆盖（tool description overrides），改写工具描述以适配不同模型的理解习惯；④ 排除指定工具或中间件，避免特定模型加载不兼容的组件；⑤ 注入额外中间件，为特定模型添加专属功能支持；⑥ 编辑自动添加的通用子智能体，优化子智能体与主模型的适配性。

3\. 运行流程：① 开发人员通过 deepagents 提供的 HarnessProfile 类创建配置文件，定义针对特定模型的专属配置；② 通过 register\_harness\_profile 方法注册配置文件，关联对应的模型标识（如“openai:gpt\-5\.5”）；③ 创建 deepagents 智能体时，指定要使用的模型（如 gpt\-5\.5）；④ 智能体初始化过程中，自动检测该模型对应的已注册配置文件，并将配置内容应用到智能体中；⑤ 智能体运行时，将按照配置文件中的设置（如系统提示词后缀、工具描述）执行任务，适配对应模型的特性；⑥ 若更换智能体使用的模型，智能体会自动加载新模型对应的配置文件，无需手动修改智能体代码。

4\. 核心用途：适用于多模型适配场景，当智能体需要切换不同大模型运行时，通过配置文件可快速适配各模型的特性，避免重复编写适配代码；同时可针对不同模型的优势与短板，定制专属配置（如为 Claude 优化指令遵循能力，为 GPT 优化工具调用准确性），提升智能体在不同模型下的运行效果。

5\. 配置与使用示例（贴合 deepagents 代码规范）：通过导入 HarnessProfile 和 register\_harness\_profile 方法，可快速创建并注册配置文件，以下为核心示例：

从 deepagents 库中导入相关类和方法：from deepagents import HarnessProfile, register\_harness\_profile；

注册针对特定模型的配置文件（以 gpt\-5\.5 为例）：当选择 gpt\-5\.5 模型时，自动为系统提示词追加后缀“Respond in under 100 words\.”，约束模型输出长度。

register\_harness\_profile\(
    "openai:gpt\-5\.5",
    HarnessProfile\(system\_prompt\_suffix="Respond in under 100 words\."\),
\)

6\. 关联概念：Provider Profiles（提供商配置文件）是 Profiles 的轻量化配套 API，主要用于封装特定模型提供商的模型构建参数（如 API 密钥、超时设置、重试规则），与 Harness Profile 协同工作，Harness Profile 负责模型运行时的行为配置，Provider Profiles 负责模型构建时的基础参数配置，共同提升多模型、多提供商场景下的配置效率。

7\. 与其他组件的协同关系：配置文件的配置内容会作用于智能体的核心运行环节，与技能、记忆、各类后端协同工作——配置文件中的系统提示词后缀可约束技能调用、记忆加载的相关输出；工具描述覆盖可优化工具调用的准确性，适配不同模型对工具描述的理解习惯；中间件注入可新增与后端交互的专属功能（如优化沙箱后端与特定模型的适配性），进一步提升智能体的整体运行效果。

## Structured output（结构化输出）

在 deepagents 生态中，结构化输出是智能体的核心功能之一，支持将模型生成的内容按照指定的规范格式输出，实现输出结果的标准化、可解析化，便于智能体与其他系统、组件协同工作，同时提升结果的可用性与可验证性。

1\. 核心作用：让智能体按照预设的结构化模式生成输出内容（如 JSON、XML 等格式），替代无规则的自然语言输出，解决模型输出结果难以被程序解析、验证的问题；同时通过内置的验证机制，确保输出内容符合预设规范，减少格式错误导致的流程中断，适配智能体与外部系统对接、数据自动化处理等场景。
