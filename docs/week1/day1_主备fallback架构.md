## 一、为什么企业级 Agent 必须有主备

### 1. LLM API 的可用性现实

大模型 API **不是** 100% 可用的。生产环境常见的失败场景：

| HTTP 码 | 含义 | 例子 | 性质 |
|---------|------|------|------|
| 429 | 限流 | DeepSeek 高峰期 QPS 超限 | 瞬时（可能恢复） |
| 500/502/503 | 服务端故障 | provider 内部错误、网关超时 | 瞬时或持久 |
| 401 | 鉴权失败 | key 泄露被吊销、配置错误 | 持久 |
| 402 | 余额/权益不足 | Kimi 会员过期、账户欠费 | 持久 |
| 408/超时 | 网络抖动 | 跨境访问、DNS 问题 | 瞬时 |
| 400 | 请求错误 | 上下文超限、参数错误 | 持久（需改请求） |

**关键认知**：单 provider 的可用性通常在 99%~99.9%，意味着一年有 8~87 小时不可用。企业 SLA 要求 99.95%+，单点根本扛不住。

### 2. 单点依赖的三类风险

- **可用性风险**：provider 故障 → 业务全线中断
- **能力风险**：某 provider 不支持 Function Calling / 某工具 → Agent 闭环断掉
- **供应商锁定**：被一家涨价/下线模型绑架，无议价能力

> **面试加分点**：主备不只是"容灾",还是**议价筹码**和**能力互补**——DeepSeek 性价比高当主力，豆包响应快做备用，Claude 推理强处理难题。

## 二、五种核心容灾模式

这是面试重灾区。要能说清每种模式解决什么问题、什么时候用。

### 模式 1：重试（Retry）

**解决**：瞬时故障（网络抖动、临时 429）。
**做法**：失败后等一段时间重试，通常**指数退避**（exponential backoff）—— 1s → 2s → 4s → 8s。
**关键**：只对**瞬时故障**重试；401/402 这种持久故障重试 100 次也没用。
**进阶**：加抖动（jitter）防止多个客户端同时重试引发"惊群"。

```python
# 伪代码
for attempt in range(3):
    try:
        return call_llm(...)
    except TransientError:
        time.sleep(2 ** attempt)  # 1s, 2s, 4s
```

### 模式 2：主备（Active/Standby）⭐ 我们实现的就是这个

**解决**：持久故障（provider 挂了、key 失效）。
**做法**：主 provider 优先，失败自动切备用 provider，链式向下。
**关键**：切换是**不可恢复**的——主 provider 后来恢复了，当前请求也不会切回去（下次请求再从主开始）。
**代码对照**：`call_with_fallback()` 正是主备链。

```
DeepSeek(主) ──失败──> 豆包(备1) ──失败──> Kimi(备2) ──失败──> 报错
```

### 模式 3：多活 / 负载均衡（Active/Active）

**解决**：高并发下的吞吐 + 可用性。
**做法**：多个 provider 同时承接流量，按权重/延迟分流，故障节点自动摘除。
**与主备区别**：主备是"主忙活、备歇着"，多活是"大家一起干"。
**适用**：高 QPS 场景；训练项目用不到，但要知道。

### 模式 4：熔断（Circuit Breaker）

**解决**：防止故障雪崩。
**做法**：连续失败 N 次后**短路**，一段时间内不再请求该 provider（直接失败或走备用），定期"半开"探测是否恢复。
**为什么需要**：主备链里如果主 provider 挂了，每个请求都要先试主、等超时、再切备——大量请求堆积会把系统拖垮。熔断让主"快速失败"，直接走备用。
**三态**：Closed（正常）→ Open（熔断，直接拒绝）→ Half-Open（探测恢复）。

> **面试高频**：主备 + 熔断通常一起用。我们当前代码**没有熔断**，是个可改进点（面试时主动提"当前实现缺熔断，生产环境应加"是加分项）。

### 模式 5：降级（Graceful Degradation）

**解决**：能力故障（不是 provider 挂了，是能力不够）。
**做法**：强模型 → 弱模型 → 规则/缓存 → 兜底文案。
**例子**：Claude 不支持某工具 → 退到 DeepSeek → 还不行 → 用规则引擎 → 最后返回"暂时无法处理"。

```
强模型(Claude) ──不支持tool──> 弱模型(DeepSeek) ──失败──> 规则引擎 ──> 兜底文案
```

## 三、切换策略的四个维度

面试官常问"还能怎么切"。要能从四个维度展开：

| 维度 | 含义 | 例子 |
|------|------|------|
| **Provider 级** | 换厂商 | DeepSeek → 豆包 → Kimi |
| **Model 级** | 同厂商换模型 | deepseek-v4-flash → deepseek-reasoner |
| **Capability 级** | 按能力降级 | 支持 tool 的模型 → 不支持 tool 的纯文本 |
| **Region 级** | 换地域端点 | 火山北京区 → 火山上海区 |

我们当前实现的是 **Provider 级**。

## 四、设计要点（面试题精选）

### Q1：重试和切换怎么选？
看**错误码**：
- 429/5xx/超时 → 瞬时，**先重试**（2-3 次），还失败再切换
- 401/402/404 → 持久，**直接切换**，不浪费重试次数
- 400 → 请求本身有问题，**改请求**，重试切换都没用

**最佳实践**：重试 + 切换组合——每个 provider 内部先重试 2 次，再切下一个 provider。

### Q2：fallback 链多长合适？
**2-3 个**足够。
- 太短：容灾能力不足
- 太长：尾部 provider 能力差、成本难控、延迟累积
- 生产常见：1 主 + 1-2 备

### Q3：不同 provider 输出风格不一致怎么办？
这是**真实痛点**。DeepSeek 和豆包对同一 prompt 的回答格式、用词都不同。
**对策**：
- System Prompt 里用 **JSON Schema / 严格格式约束**强制输出结构
- 关键字段做后处理校验，不合规就重试或切备用
- 评测：每个 provider 跑同一批测试用例，知道差异边界

### Q4：重试的幂等性问题？
LLM 文本生成**天然幂等**（同样的输入产生文本，没有副作用）。
但**如果 Agent 在循环里调工具**，重试可能导致工具被调用多次——比如"发邮件"工具重试会发两封。
**对策**：工具调用做幂等键（idempotency key），服务端去重。

### Q5：成本怎么控？
- 主用**便宜**的（DeepSeek ¥1/百万 Token）
- 备用可以**贵但强**的（应急用，流量小）
- 监控每个 provider 的调用量和成本，设预算告警
- 加**缓存**：相同请求命中缓存，不调 API

### Q6：怎么知道该切了？（监控）
- **成功率**：滑动窗口内某 provider 成功率低于阈值 → 降权或熔断
- **延迟**：P99 延迟突增 → 可能要切
- **错误码分布**：401 突增 → key 可能被吊销

## 八、面试场景图解（5 种容灾模式流程图）

> 面试时能画图讲解 = 从"知道概念"升级到"能讲清楚"。下面对每种模式画出请求流转图。

### 模式 1：重试（Retry）

```
请求 ──> [Provider]
          │
       成功? ──是──> 返回
          │
          否（瞬时故障 429/5xx/超时）
          │
       等待 2^n 秒（指数退避 + 抖动）
          │
       重试（最多 3 次）──仍失败──> 走下一模式（切换）
```

**口诀**：瞬时故障重试，指数退避加抖动。

### 模式 2：主备（Active/Standby）⭐ 我们实现的

```
请求 ──> [DeepSeek 主]
          │
       成功? ──是──> 返回
          │
          否（任意错误）
          │
       [豆包 备1]
          │
       成功? ──是──> 返回
          │
          否
          │
       [Kimi 备2] ──失败──> 报错（所有 provider 不可用）
```

**口诀**：持久故障切换，链式向下，第一个成功即返回。
**注意**：切换是请求级的——这次请求切到豆包，下次请求还是从 DeepSeek 开始（主恢复后自动回切）。

### 模式 3：多活（Active/Active）

```
                    ┌──> [DeepSeek]  权重 60%
请求 ──> [负载均衡] ─┼──> [豆包]      权重 30%
                    └──> [Kimi]      权重 10%
                          │
                  健康检查摘除故障节点
```

**口诀**：多节点同时承接，按权重分流，故障自动摘除。
**与主备区别**：主备是"主忙备歇"，多活是"大家一起干"。高 QPS 场景用。

### 模式 4：熔断（Circuit Breaker）

```
                ┌──────────────────────────┐
                │   连续失败 ≥ N 次？       │
                └──────────────────────────┘
                    │ 是                │ 否
                    ▼                   ▼
            ┌──────────────┐     正常请求 [Provider]
            │  Open 熔断   │
            │  直接拒绝    │
            │  （走备用）  │
            └──────┬───────┘
                   │ 等待冷却时间（如 30s）
                   ▼
            ┌──────────────┐
            │ Half-Open    │
            │ 放一个探测请求│
            └──────┬───────┘
          成功? ──是──> Closed（恢复正常）
          失败 ──> 重新 Open
```

**三态口诀**：Closed 正常 → Open 熔断 → Half-Open 探测。
**为什么需要**：主 provider 挂了，每个请求都试主、等超时、再切备——堆积请求拖垮系统。熔断让主"快速失败"，直接走备用。

### 模式 5：降级（Graceful Degradation）

```
请求 ──> [强模型 Claude，支持 tool]
          │ 不支持 tool / 失败
          ▼
        [弱模型 DeepSeek，纯文本]
          │ 失败
          ▼
        [规则引擎 / 缓存]
          │ 未命中
          ▼
        [兜底文案："暂时无法处理，已转人工"]
```

**口诀**：能力故障降级，强→弱→规则→兜底，保证不崩。
**与切换区别**：切换是"换个同类"，降级是"换个弱的或退回规则"。

### 组合使用（生产级架构）

```
请求
  │
  ▼
[熔断器] ──Open──> 直接走备用
  │ Closed
  ▼
[Provider 主]
  │
  ├─ 瞬时故障 ─> [重试 2-3 次，指数退避]
  │                  │ 仍失败
  │                  ▼
  ├─ 持久故障 ─> [Provider 备]（主备切换）
  │                  │ 失败
  │                  ▼
  └────────────> [降级：规则/缓存/兜底]
```

**面试一句话**："生产级架构是熔断 + 重试 + 主备 + 降级的组合拳：熔断防雪崩，重试扛瞬时，主备扛持久，降级保兜底。"

## 九、进阶实战（生产级开源项目扩展阅读）

> 知道原理后，看生产级项目怎么实现，能快速提升认知。以下是主流的多 provider 路由/网关项目，课后自行了解。

### 1. LiteLLM（最推荐入门）

- **定位**：统一 API 网关，100+ provider 用同一套 OpenAI 协议调用
- **核心能力**：
  - 主备 fallback（`router.py` 的 `allow_fallbacks`）
  - 重试 + 指数退避（`num_retries`）
  - 限流、成本追踪、缓存
  - 熔断（cooldown 机制）
- **语言**：Python
- **看什么**：`litellm/router.py` 的 fallback 链实现，和我们 `call_with_fallback()` 思路一致，但工程化更完整
- **官网**：github.com/BerriAI/litellm

### 2. OpenRouter（SaaS 网关）

- **定位**：商用 LLM 路由服务，一个 key 调用所有模型
- **核心能力**：
  - 自动 fallback（`provider order` 配置）
  - 按价格/延迟/质量路由
  - 统一计费
- **怎么用**：不用自建，直接调 `openrouter.ai/api/v1`，配置 provider 优先级
- **学什么**：看它的路由策略设计——价格优先、延迟优先、质量优先怎么切换
- **官网**：openrouter.ai

### 3. Portkey（企业级网关）

- **定位**：生产级 LLM 网关，强调可观测性和治理
- **核心能力**：
  - 主备 + 重试 + 超时（一行配置）
  - 请求级监控、trace、日志
  - 缓存、限流、 Guardrails（输出安全过滤）
- **学什么**：它的"治理"维度——成本控制、审计、合规，企业落地必考虑
- **官网**：portkey.ai

### 4. 字节 EinoLLM / 阿里 Spring AI Alibaba

- **定位**：国内厂商的企业级 LLM 应用框架
- **核心能力**：Agent 编排 + 多模型路由 + 可观测，生态完整
- **学什么**：国内生产环境的实战模式，比海外项目更贴合国内 provider（豆包、通义、Kimi、DeepSeek）
- **适用**：第三周写 Agent 时回来细看

### 5. 自研 vs 用网关的取舍

| 维度 | 自研（如我们当前） | 用网关（LiteLLM 等） |
|------|------------------|-------------------|
| 上手速度 | 快，几行代码 | 需学配置 |
| 灵活度 | 完全可控 | 受框架约束 |
| 生产完备度 | 低（缺熔断/监控） | 高（开箱即用） |
| 适用阶段 | 学习、小项目 | 生产、中大型项目 |

**建议路径**：
1. **Day 1~3**：自己写主备（理解原理，我们正在做）✅
2. **Day 4+**：读 LiteLLM 源码，看生产级怎么加熔断/重试/监控
3. **第三周写 Agent**：评估是否直接用 LiteLLM 做底层网关

> **面试加分**："我训练时先手写主备理解原理，再读 LiteLLM 看生产级实现。知道为什么自研、什么时候该上开源，比只会调库更有价值。"
