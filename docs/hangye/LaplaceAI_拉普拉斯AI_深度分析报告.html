1.2 战略定位：AI原生工作流编排平台

拉普拉斯AI的自我定位是"AI原生智能生产力操作平台"，核心理念可概括为AI-PaaS（AI Productivity as a Service，AI生产力即服务）。与传统SaaS/RPA路线不同，拉普拉斯不向现有流程中"植入"AI工具，而是以AI OS为底层思路，让AI去思考如何调用不同的应用来完成任务。

核心差异化：不改造企业现有系统，而是构建一个AI编排层——让Agent"理解"业务意图，自动编排ERP/OA/MES等现有系统和AI工具，完成端到端的工作流。创始人衣冠锡将此比喻为"软件3D打印"：基于不同场景，根据手里已有的工具和素材，动态组合出当时所需的解决方案，而非为每个需求开发一套新软件。

4.2 Agent编排核心：Agent 0 → 专家Agent → 工具

拉普拉斯的Agent编排采用分级架构，其核心设计是：

  • Agent 0（领导Agent）：最顶层的调度Agent，理解任务意图，决定"谁来做"，负责唤醒和交接
  • 专家Agent：按功能分类的专业Agent，每个负责特定领域（如信息收集、信息处理、信息分发）
  • 工具（自身也可能是Agent）：底层工具可以是API、模型、第三方服务，也可被封装为Agent

这种"领导-专家-工具"的三级结构，让虚拟员工可以领导调动虚拟员工，形成"AI团队"协作模式。

4.3 信息三层抽象：降低交互复杂度

为解决多Agent协作中交互维度爆炸的问题，拉普拉斯将所有业务抽象为三层：

层级功能标准化内容典型场景
信息收集 统一从各类渠道获取输入 IM、邮件、CRM、后台数据 — 标准化接入 微信/企业微信消息抓取、客户数据查询
信息处理 中间层的数据加工与转换 文字总结、思维导图生成、PPT制作 — 标准化流程 简历分析、舆情监控、报告生成
信息分发 将处理结果发送到目标渠道 输出到IM、邮件、Excel/Word — 标准化交付 日程文件发送回微信、报告分发

设计哲学：创始人强调，底层抽象越简洁，上层出错的概率越低。"如果底层设计得越复杂，往上走的时候出错概率越高。市场需求越多，就越考验你的归类和交互能力。"这种三层抽象让不同行业的业务逻辑（如化妆品研发vs新能源研发）共享相同的技术底层——都是抓取竞品材料、结合现有产品、进行调试、输出到Excel/Word。

4.4 LKHM知识库：Know-How沉淀引擎

LKHM（Laplace Know-How Memory）是拉普拉斯自研的知识沉淀系统，本质是一个不断完善的Agent编排模板库。它通过四种方式积累知识：

方式机制效果
RAG（最优） 从对话和操作中自动收集增量知识，存入向量库 实时学习用户偏好，无需刻意准备
Prompt工程 改良Prompt并进行版本控制 逐步优化Agent行为
增量预训练 持续向大模型注入领域知识 提升专业领域理解能力
微调 针对用户习惯和意图识别做个性化微调 适配不同用户的表达方式和企业流程差异

4.5 Graph结构编排：超越思维链

创始人明确指出，拉普拉斯的Agent协作逻辑不是线性思维链（A→B→C），也不是简单的网络结构，而更接近Graph（图结构）。这意味着：

  • Agent之间的关系是多对多的——一个工具Agent可能在某场景下被Agent A调用，在另一场景下被Agent B调用
  • 执行过程是异步并行的——多个Agent可以同步进行，不是线性串行
  • Token成本因此不会倍增——大部分操作不一定需要消耗大模型token

五、AI OS理念与增强型MCP协议

5.1 "AI OS"概念解读

拉普拉斯的"AI OS"并非传统操作系统（管理CPU/内存/IO），而是面向Agent的运行时管理层。与传统OS的对比：

维度传统OSLaplace AIOS / Agentic OS
调度对象线程、进程Agent及其原子动作（LLM推理、工具调用）
调度目标公平性、吞吐量目标完成效率、关键路径推进
调度算法时间片轮转、CFS目标驱动调度、DAG感知调度
内存管理虚拟内存页向量化记忆、情景记忆检索
通信机制管道、消息队列语义化通道、协作合同、意图路由
安全模型DAC/MAC/UIDCapability Token、动态权限
工具接入设备驱动MCP驱动，统一接入API/数据库/SaaS
错误处理信号、core dump目标修复、自动回滚、备用Agent接管

5.2 增强型MCP协议

拉普拉斯采用增强型MCP（Model Context Protocol）协议实现智能体间能力互通。相比传统REST API，MCP的核心优势：

对比维度REST API增强型MCP
能力发现手动配置内核自动发现
权限管理各系统独立管理Runtime统一治理
审计能力分散系统级统一审计
通信模式请求-响应支持异步与流式
Agent支持非原生AI-native
系统集成应用级Runtime / OS级

6.3 优先场景选择逻辑

CEO衣冠锡在访谈中分享了选择优先场景的经济逻辑：

  • 优先场景：信息密集型服务业（旅行咨询/留学咨询/猎头）、金融（分析师写报告）、PR舆情监控——这些场景信息量大、人力成本高、容错度尚可
  • 不优先：行政岗位——虽然可替代但商业价值有限（"行政工资偏低，多一个少一个不在乎"）；硬核研发和律师——对信息本质质量要求极高，容错度低
  • 核心判断："在行业里需要找在对方看起来也比较有价值的事情"——替代工资高的分析师比替代低薪行政更有商业价值

七、商业模式与生态策略

7.1 商业模式：企业智能化合伙人

拉普拉斯不采用传统SaaS的订阅制或RPA的项目制，而是定位为"企业智能化合伙人"，通过AI-PaaS模式深度嵌入客户业务：

  • 标准化平台 + 模块化定制：底层框架标准统一，不同场景只是"接上去的工具不一样"，核心研发动作没有区别——类比"不同职业的人用同一部iPhone，装不同APP"
  • 流程沉淀复用：AI在工作中不断积累流程模板（LKHM），形成可复用、可分享的"编排模板库"
  • 团队协作机制：支持组织内流程智能分享，提高团队协作效率

7.2 生态策略：共生而非竞争

策略维度具体做法
与IM平台合作 与句子互动（Wechaty）合作——对方有IM入口和飞书/企业微信通道，拉普拉斯提供智能化处理能力，1+1>2
不抢别人生意 "我们尽可能共生，不去抢别人的生意也不去嫖"——不做IM、不做大模型，专注编排
工具生态 优先使用市面上已有的好工具（HuggingFace模型、第三方API），只在必要时自建
开发者生态 计划建立AI共创社区，让开发者按规则开发应用、用户安装使用（非GPT Store模式）
出海策略 先做国内再出海——国内场景丰富、数据获取容易；海外对数据侵入性顾虑大（如三星不让员工用GPT）

7.3 定制化vs标准化

创始人明确表示："实质上还是标准化"。不同行业和客户的核心差异在于"接入的工具不一样"和"流程先后顺序不同"，但底层逻辑统一——信息从哪来、中间怎么处理、最后分到哪去。拉普拉斯的策略是"先做不同场景，找到共同点"，逐步实现模块化。

9.2 核心差异化总结

拉普拉斯AI = 编排型（Orchestrate）——以AI OS思路管理Agent调度和工具编排，核心差异化在于"旧系统兼容联动"。这是所有9家对标公司中唯一以"不替换企业现有系统"为核心卖点的能力。其他公司要么替换（来也RPA替换手工流程）、要么新建（未来式建数字岗位）、要么嵌入（语核嵌入知识检索），而拉普拉斯选择的是"在旧系统之上建编排层"——这条路线最接近企业现实痛点，但也面临最大的工程复杂度挑战。

十、总结与展望

10.1 优势

  • 旧系统兼容是杀手级能力：大多数AI Agent平台要求企业迁移到新系统，拉普拉斯的"AI直接接上就能用"直击企业最大顾虑——保护既有IT投入
  • 经济学博士的系统性思维：CEO衣冠锡从信息流转和组织效率角度看AI，而非纯技术视角，这让他能洞察"替代行政不如替代分析师"的商业价值逻辑
  • Agent编排而非Agent构建："尽量不做Agent，专注编排"——这在Agent生态碎片化的当下是明智的定位，避免了与大模型厂商正面竞争
  • Graph结构编排领先：超越线性的思维链，采用异步并行Graph结构，理论效率更优
  • 信息三层抽象降低复杂度：将所有业务抽象为"收集→处理→分发"，大幅降低多Agent协作的交互出错概率
  • 互补型生态策略：与句子互动等IM入口合作，"不抢别人生意"，降低树敌风险

10.3 发展展望

拉普拉斯AI处于一个赛道正确但时机超前的位置。"AI OS"和"Agent编排"是2026年行业最热的概念方向之一，但真正能落地的企业级产品寥寥无几。拉普拉斯的"旧系统兼容"理念精准命中了企业AI落地的最大痛点——不换系统就能用AI，这比"建一个全新的AI原生系统"务实得多。

但11人团队、天使轮融资的体量，要支撑一个"多智能体操作系统"的产品愿景，面临的是与微软、Google级别资源的正面竞争。其生存策略可能是被大厂生态收购/集成，或在垂直制造业场景中找到足够深的护城河（如智能排产），以"小而美"的方式存活。

一句话总结：拉普拉斯AI是AI Agent赛道中最具"操作系统思维"的早期初创公司——以AI编排重构企业工作流，以旧系统兼容为差异化壁垒，以经济学博士的系统性视角选择场景——但团队和资金体量与产品愿景之间的巨大落差，是其最大的生存挑战。
