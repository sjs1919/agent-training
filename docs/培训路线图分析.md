## 前置摘要：Vibe Coding 基础 → 企业级 Agent 开发 · 必学知识块

> **Vibe Coding 基础假设**：已能通过 AI 对话生成代码、写过 Prompt、有基本编程能力（Python/JS）。以下按"已有 Vibe Coding 基础"的学习者视角重新排序，只列必须完成的核心知识块。

| 序号 | 知识块 | 来源图 | 为什么必须学 | Vibe Coder 的认知跃迁 |

| 2 | **Agent 核心机制** | 图1、图2、图5、图6 | 企业级 Agent 开发的本质。ReAct 循环、工具调用、记忆管理是单 Agent 的三根支柱 | 从单轮问答 → 多轮自主决策。核心：ReAct 模式、Plan-Execute、上下文窗口管理、Agent 记忆（短期/会话/长期） |
| 3 | **Tools & Skills 体系** | 图2、图3 | Agent 的"手和脚"。没有工具，Agent 只能聊天不能干活 | 从纯文本输出 → Agent 执行真实操作。核心：工具注册与 Schema 定义、错误处理、权限边界、调用链追踪 |

### 学习路径建议（P0 优先，线性推进）

```
Vibe Coding 基础
    │
    ▼
┌──────────────────────────────────────┐
│ P0：Agent 开发核心能力（约 2 周）      │
│ API → Agent 机制 → Tools → MCP → RAG │
│ 产出：一个能调用工具的单 Agent Demo    │
└──────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────┐
│ P1：工程化与系统化（约 2 周）          │
│ 编排 → 提示词工程 → 集群 → 鉴权 → 观测 │
│ 产出：可上线的多 Agent 系统            │
└──────────────────────────────────────┘
    │
    ▼

└──────────────────────────────────────┘
```

> **关键原则**：P0 不完成，不进 P1。P0 是从 Vibe Coder 到 Agent Developer 的质变门槛——这 5 个知识块构成了 Agent 开发的"最小可行知识集"。

## 三、总结

1. **前 5 张图偏向"能力图谱"**：更接近企业内部培训体系设计——按能力层级划分，强调工程落地（业务匹配、MCP、鉴权、可观测性），Agent 方向深度更深（集群、跨 Agent 记忆）。

2. **第 6 张图偏向"学习计划表"**：面向个人自学者的分阶段路线图，特点是**时间可执行**（总计约 2 个月），覆盖面更广（LlamaIndex、Transformer、PEFT、多模态），但工程落地维度较弱，缺少 MCP、可观测性、鉴权等企业级关注点。

3. **互补关系**：第 6 张图的时间规划可作为前 5 张能力图谱的**执行节奏参考**；前 5 张图的 MCP/鉴权/可观测性/业务匹配等内容可补充第 6 张图缺失的**工程落地维度**。

| **重要章节摘要** | **Schema 设计**：description 是给模型看的"使用说明"，写得越清晰模型调用越准确，给正负示例；**工具粒度**：一个工具只做一件事，不要设计"万能工具"；**错误处理**：工具返回 error 时模型会自动重试或换策略，错误信息要结构化（含 error_code + suggestion）；**权限模型**：每个工具标注所需权限级别，执行前校验；**调用链**：Agent → 工具选择 → 参数填充 → 执行 → 结果注入 → 下一步决策，全程打点 |
| **跳过** | 平台特定 SDK 的工具注册语法（概念通用，具体 SDK 查文档） |

| **中文资料** | [vLLM 性能优化实战 — 阿里云](https://developer.aliyun.com/article/1686141) ⭐ · [KV-Cache 调优 10 大方法 — 阿里云](https://developer.aliyun.com/article/1684500) · [LMCache：KV 缓存复用 — 腾讯云](https://cloud.tencent.cn/developer/article/2607770) · [vLLM v1 版本重大更新 — HyperAI](https://hub.baai.ac.cn/view/42378) · [AIBrix KVCache 多级卸载 — InfoQ](https://www.infoq.cn/article/qpfyamoffpda4exx5fve) |
| **必须章节** | ① PagedAttention 原理（类 OS 虚拟内存管理 KV 缓存，接近零浪费） ② 前缀缓存（Prefix Caching）：相同前缀（System Prompt/Few-shot）跨请求复用 KV 页，零成本加速 ③ 连续批处理（Continuous Batching）：动态合并请求，吞吐 2-4x ④ KV Cache 量化（FP8）：显存减半，批次翻倍 ⑤ 推测解码（Speculative Decoding）：小模型先猜，大模型验证，加速 2.5x ⑥ 生产配置核心参数 |
| **重要章节摘要** | **PagedAttention**：将 KV 缓存切分成固定大小的 Block（页），非连续存储，按需分配，内存利用率从 20-40% 提升到接近 100%；**前缀缓存条件**：基于 Block 级别匹配（一个 token 不同就失效），System Prompt 标准化+固定 Few-shot 模板可显著提升命中率；**生产推荐配置**：`vLLM + AWQ 量化 + FP8 KV Cache + 前缀缓存`，Qwen2.5-7B 在 A100 上可达 128 并发；**前缀缓存前提**：高重复度的 System Prompt 场景（RAG/多轮对话）有效，随机数据场景几乎无效；**vLLM v1 改进**：吞吐量比 v0 提升 1.7x，引入隔离式 EngineCore+统一调度器+零开销前缀缓存；**分块预填充**：将长 Prompt 拆分成小块与 Decode 交替执行，减少排队延迟；**LMCache**：多级 KV 存储（GPU→CPU→磁盘→分布式），TTFT 优化 3-10x |
| **跳过** | TensorRT-LLM 等 NVIDIA 专属方案（vLLM 更通用）、FlashAttention 内核实现细节 |
