## P0 · ⭐ 必补（今天，与 week3 代码直接相关）

### 1. LangGraph Reducer 与 `add_messages`

**为什么补**：`langgraph_agent.py:277` 的 `AgentState` 用的是裸 `messages: list[dict]`，**没有** `Annotated[list, add_messages]` reducer。代码靠每个节点返回完整 state 手动管理累积--能跑，但没用到 LangGraph 的核心状态合并机制。理解 reducer 才知道 LangGraph 为什么比手写 while 循环强（多节点并发写同一字段的合并规则）。

**代码对应**：`AgentState` 定义 + 各节点 `return {**state, "messages": state["messages"] + [...]}` 的手动拼接。

**章节 + 链接**：
- LangGraph Concepts · Low-level · **State / Reducers / add_messages** ⭐
  https://langchain-ai.github.io/langgraph/concepts/low_level/#reducers
- How-to · State Management（TypedDict + Annotated 写法）
  https://langchain-ai.github.io/langgraph/how-tos/state-management/

> **读完对照**：把 `AgentState.messages` 改成 `Annotated[list, add_messages]`，节点就能只返回新增消息（`{"messages": [new_msg]}`），LangGraph 自动 append--理解了 reducer 就明白这省了什么。

### 2. LangGraph `ToolNode` + prebuilt `create_react_agent`

**为什么补**：week3 的 `select_and_execute` 节点是**手写**的工具执行器（解析 tool_calls -> 分派 -> 拼 tool role 消息）。LangGraph 有内置 `ToolNode` 和预制的 `create_react_agent`，做的是同一件事但更标准。走读时你手写了一遍，现在该看"官方怎么写"，才能评判何时该用预置、何时该手写。

**代码对应**：`select_and_execute` 节点（手写版）vs `langgraph.prebuilt.ToolNode` / `create_react_agent`。

**章节 + 链接**：
- How-to · Use ToolNode for tool calling ⭐
  https://langchain-ai.github.io/langgraph/how-tos/tool-calling/
- Reference · Prebuilt components（ToolNode / ToolsCondition / create_react_agent）
  https://langchain-ai.github.io/langgraph/reference/prebuilt/
- Tutorial · Build a ReAct Agent（官方版 week3 Agent）
  https://langchain-ai.github.io/langgraph/tutorials/introduction/#part-3-add-tools

> **读完对照**：用 `create_react_agent(model, tools)` 一行替代 `build_graph()` 的几十行--理解官方抽象后，回头看你手写的版本哪些是"教学需要"、哪些是"生产没必要手写"。

---

### 3. LangGraph Checkpointer（持久化与记忆）

**为什么补**：week3 的 Agent 每次 `run_agent` 都是全新状态，**没有记忆**--关掉进程对话就丢。生产的 Agent 必须能"记住上次聊到哪"，这靠 Checkpointer。代码完全没用，是走向生产的第一道缺口。

**代码对应**：`build_graph()` 里 `graph.compile()` **没传 checkpointer**--所以无持久化。

**章节 + 链接**：
- Concepts · Persistence（Checkpointer / thread_id / 内存 vs SqliteSaver） ⭐
  https://langchain-ai.github.io/langgraph/concepts/persistence/
- How-to · Persist state（MemorySaver / SqliteSaver 实操）
  https://langchain-ai.github.io/langgraph/how-tos/persistence/

> **读完对照**：给 `compile(checkpointer=MemorySaver())` 加一行 + 调用时传 `thread_id`，Agent 就能跨调用记上下文。这是 week3 -> week4 多 Agent 记忆的基础。

### 5. MCP Streamable HTTP transport

**为什么补**：week3 只用 stdio（本地子进程）。生产部署 MCP Server 给多 Client 共享、远程访问，必须用 Streamable HTTP。走读时你只体验了 stdio 这条路。

**代码对应**：`order_server.py` / `resource_server.py` 都 `mcp.run(transport="stdio")`；`test_mcp_client.py` 用 `stdio_client`。

**章节 + 链接**：
- MCP 规范 · Transports（stdio / Streamable HTTP，SSE 已弃用） ⭐
  https://modelcontextprotocol.io/specification/2025-03-26/basic/transports
- MCP Python SDK · writing-http-server（示例）
  https://github.com/modelcontextprotocol/python-sdk

> **读完对照**：把 `order_server` 改成 `mcp.run(transport="streamable-http")`，Client 用 `streamablehttp_client` 连--理解远程 MCP 的部署形态。

### 6. MCP Resources / Prompts 两个原语

**为什么补**：MCP Server 有三种能力（Tools / Resources / Prompts），week3 只用了 Tools。Resources（暴露数据，应用可控）和 Prompts（预定义模板，用户可控）是另两条腿，合同/订单场景里"只读数据"该用 Resources 而非 Tools。

**代码对应**：`order_server.py` 全是 `@mcp.tool()`，没有 `@mcp.resource()` / `@mcp.prompt()`。

**章节 + 链接**：
- MCP 规范 · Server Resources ⭐
  https://modelcontextprotocol.io/specification/2025-03-26/server/resources
- MCP 规范 · Server Prompts
  https://modelcontextprotocol.io/specification/2025-03-26/server/prompts
- MCP 中文站 · 核心概念 -> 资源 / 提示词
  https://mcpcn.com/

> **读完对照**：订单 CSV 这种"只读数据"用 `@mcp.resource("orders://list")` 暴露比用 tool 更语义化--理解三原语的边界。
